# Niagara 整体框架与设计理念

## Niagara 的诞生背景

UE4 原有的粒子系统是 **Cascade**。Cascade 的核心问题：

1. **固定管线**：模块是 C++ 硬编码的，美术想加新行为必须程序员写代码
2. **不可组合**：模块之间的数据流是隐式的，难以自由组合
3. **难以扩展**：加一个新的数据源（比如从骨骼网格采样）需要改引擎代码

Niagara 的设计目标就是解决这三个问题：**让美术/TA 能自由编写粒子行为，而不需要改 C++ 代码。**

---

## 核心设计理念

### 1. 数据驱动，而非行为驱动

Cascade 的思维方式是："我有一个 Spawn Rate 模块、一个 Initial Velocity 模块"——每个模块是一个**行为**。

Niagara 的思维方式是："我有一组**属性**（Position, Velocity, Color...），模块是对这些属性的**读写操作**"。

```
Cascade:  模块 = 黑盒行为（输入隐式，输出隐式）
Niagara:  模块 = 对命名属性的显式读写函数
```

这就是为什么 `NiagaraParameterStore` 和命名空间（`Particles.Position`、`Emitter.SpawnRate`）如此重要——它们是整个系统的数据总线。

### 2. 三级层次结构

```
System（系统）
  ├── 有自己的属性和脚本（SystemSpawn / SystemUpdate）
  │
  ├── Emitter A（发射器）
  │     ├── 有自己的属性和脚本（EmitterSpawn / EmitterUpdate）
  │     └── Particle（粒子）
  │           └── 有自己的属性（Position, Velocity, Age...）
  │               被 ParticleSpawn / ParticleUpdate 脚本操作
  │
  └── Emitter B
        └── ...
```

每一层都有：
- **自己的属性（Attributes）**——持久化数据，跨帧保留
- **Spawn 脚本**——初始化时执行一次
- **Update 脚本**——每帧执行

数据可以**向下流动**：System 参数可以被 Emitter 读取，Emitter 参数可以被 Particle 读取。但**不能向上**：Particle 不能直接写 System 属性（需要通过事件或 DI）。

### 3. 堆栈式执行模型（Module Stack）

每个阶段（如 Particle Update）不是一个单一脚本，而是一个**模块堆栈**：

```
Particle Update Stack:
  ├── [Module] Update Age                  ← 读 DeltaTime，写 Age
  ├── [Module] Gravity Force               ← 写 Particles.Force
  ├── [Module] Drag                        ← 读/写 Velocity
  ├── [Module] Solve Forces and Velocity   ← 读 Forces，写 Velocity
  ├── [Module] Scale Color                 ← 读 NormalizedAge，写 Color
```

所有模块**按顺序执行，共享同一组属性**。Module A 写的 `Particles.Velocity`，Module B 可以立即读到。这就是为什么模块顺序很重要。

编译时，整个堆栈被**展开合并为一个脚本**（一段 VM 字节码），不是运行时逐个调用模块。

## 四大核心子系统

### 子系统一：类型系统 + 参数系统

```
FNiagaraTypeDefinition   ← 类型（float, vec3, bool, DataInterface...）
FNiagaraVariable         ← 变量 = 名字 + 类型 + 值
FNiagaraParameterStore   ← 变量的集合（带绑定、推送机制）
```

**命名空间**决定了变量的作用域和生命周期：

| 命名空间 | 含义 | 生命周期 | 谁能读 | 谁能写 |
|---|---|---|---|---|
| `System.xxx` | 系统属性 | 跨帧持久 | 所有层 | System 脚本 |
| `Emitter.xxx` | 发射器属性 | 跨帧持久 | Emitter + Particle | Emitter 脚本 |
| `Particles.xxx` | 粒子属性 | 跨帧持久 | Particle | Particle 脚本 |
| `User.xxx` | 用户参数 | 外部设置 | 所有层 | 蓝图/C++ |
| `Engine.xxx` | 引擎参数 | 只读 | 所有层 | 引擎 |
| `Module.xxx` | 模块输入 | 当前模块 | 当前模块 | 上层绑定 |
| `Transient.xxx` | 阶段临时变量 | 当前阶段 | 当前阶段 | 当前阶段 |

**参数推送链**：
```
User Parameters (蓝图设置)
  → System ParameterStore
    → Emitter ParameterStore
      → ScriptExecution ParameterStore → VM 读取
```

### 子系统二：VM（虚拟机）

Niagara 有自己的**字节码虚拟机**，类似一个简单的 SIMD 计算机：

```
节点图 → HLSL-like 中间表示 → 编译为 VM ByteCode
```

VM 的特点：
- **基于寄存器**（不是基于栈）
- **批量执行**：一条指令同时操作多个粒子（SIMD 风格）
- **外部函数调用**：通过 `FVMExternalFunctionBindingInfo` 调用 C++ 函数（主要是 DataInterface 提供的）
- **双缓冲**：读取 CurrentData，写入 DestinationData

VM 执行的核心循环：
```cpp
for each particle batch (VECTOR_WIDTH 个粒子一组):
    for each bytecode instruction:
        execute(registers[...])  // 可能是算术、条件、外部函数调用等
```

对于 GPU Emitter，同样的节点图被编译为 **HLSL Compute Shader**，不走 VM。

### 子系统三：DataSet（数据集）

DataSet 是 Niagara 的**内存模型**：

```
FNiagaraDataSet ← 管理器（双缓冲、ID 管理）
  └── FNiagaraDataBuffer ← 一帧的数据
        ├── FloatData[]   ← SOA: 所有 float 分量紧凑排列
        ├── Int32Data[]   ← SOA: 所有 int32 分量
        └── HalfData[]    ← SOA: 所有 half 分量
```

为什么用 [[SOA与AOS|SOA]] 而不是 [[SOA与AOS|AOS]]？
- **SIMD 友好**：处理 4 个粒子的 X 坐标时，数据是连续的
- **GPU 友好**：Compute Shader 的 coalesced memory access
- **灵活属性**：不同 Emitter 可以有不同的属性集，不浪费内存

**粒子生死管理**：
- 新粒子追加到 Buffer 末尾
- 死亡粒子通过 `SwapInstances`（把最后一个粒子换到死亡位置）移除
- 可选的 Persistent ID 系统维护 ID → Index 映射

### 子系统四：DataInterface（数据接口）

DataInterface 是 Niagara 的**扩展机制**——它是脚本与外部世界的桥梁：

```
UNiagaraDataInterface (基类)
  ├── UNiagaraDataInterfaceCurve          ← 曲线采样
  ├── UNiagaraDataInterfaceSkeletalMesh   ← 骨骼网格采样
  ├── UNiagaraDataInterfaceSpline         ← 样条线
  ├── UNiagaraDataInterfaceGrid3D         ← 3D 网格
  ├── UNiagaraDataInterfaceAudio          ← 音频
  └── ...
```

DI 的工作方式：
1. **声明函数签名**：告诉编译器 "我提供了 GetPosition(Index) → Vec3 这个函数"
2. **提供 VM 实现**：注册 C++ 函数，VM 通过外部函数绑定调用
3. **提供 GPU 实现**：生成 HLSL 代码，嵌入 Compute Shader

```
模块节点图中：
  [Sample Skeletal Mesh] → GetSkinnedBonePosition(BoneIndex)
                            ↓
                VM 调用 → UNiagaraDataInterfaceSkeletalMesh::GetSkinnedBonePosition()
                GPU 调用 → 内联的 HLSL 代码
```

DI 是 Niagara 可扩展性的核心——想从任何新数据源读取数据，只需要写一个新的 DataInterface 类。

## 完整的运行时执行流程

```
每帧 Tick:

1. NiagaraWorldManager::Tick()
   └── 管理所有活跃的 SystemSimulation

2. NiagaraSystemSimulation::Tick_GameThread()
   └── 批量处理同类型的 SystemInstance

3. NiagaraSystemInstance::Tick()
   ├── 推送参数（ParameterStore 绑定链）
   ├── 执行 SystemUpdate 脚本（VM）
   │     └── 更新 System 级属性
   │
   ├── for each EmitterInstance (按依赖顺序):
   │     ├── 执行 EmitterUpdate 脚本（VM）
   │     │     └── 决定 SpawnRate, 计算 Emitter 属性
   │     │
   │     ├── Particle Spawn:
   │     │     ├── 分配新粒子空间
   │     │     └── 执行 ParticleSpawn 脚本（VM/GPU）
   │     │
   │     ├── Particle Update:
   │     │     └── 执行 ParticleUpdate 脚本（VM/GPU）
   │     │
   │     ├── Event Handling:
   │     │     └── 处理事件，可能触发更多 Spawn
   │     │
   │     └── 移除死亡粒子
   │
   └── 提交渲染数据给渲染线程

4. 渲染线程:
   └── NiagaraRenderer::CreateRenderData()
       └── 读取 CurrentData，生成 DrawCall
```

## 编辑器中的概念映射

| 编辑器中看到的 | 源码对应 |
|---|---|
| System 面板 | `UNiagaraSystem` |
| 一个 Emitter 标签页 | `FNiagaraEmitterHandle` → `UNiagaraEmitter` |
| 模块堆栈中的一个模块 | 一个 `UNiagaraScript`（Usage=Module） |
| 模块的输入参数 | `Module.xxx` 命名空间的变量 |
| "User Parameters" 面板 | `ExposedParameters`（`User.xxx`） |
| Particle Spawn/Update 区域 | 模块堆栈 → 编译为一个 `UNiagaraScript` |
| 渲染器区域（Sprite/Mesh/Ribbon） | `UNiagaraRendererProperties` 子类 |
| "Selection" 面板中的参数 | `FNiagaraVariableMetaData` + 绑定 |

## 与 Cascade 的关键区别总结

| 方面     | Cascade    | Niagara            |
| ------ | ---------- | ------------------ |
| 模块实现   | C++ 硬编码    | 节点图编译              |
| 数据流    | 隐式         | 显式命名属性             |
| 扩展方式   | 改引擎代码      | DataInterface      |
| GPU 支持 | 有限         | 完整（Compute Shader） |
| 模块交互   | 固定接口       | 共享属性读写             |
| 执行模型   | 逐模块 C++ 调用 | 编译为统一字节码/HLSL      |
| 事件系统   | 无          | 有（跨 Emitter 事件）    |
| 调试能力   | 有限         | 可查看每个属性值           |
