# NiagaraCommon.h

文件位置：`Source/Niagara/Public/NiagaraCommon.h`

这个文件是 Niagara 的**"公共词*汇表"**，定义了整个系统到处*使用的枚举、常量和基础结构体。你不需要背它，但需要知道每个东西是干什么的，后面读其他文件时会不断碰到。

---

## 一、全局常量（第 27-41 行）

```cpp
constexpr uint32 NiagaraComputeMaxThreadGroupSize = 64;       // GPU线程组最大大小
constexpr uint32 NiagaraMaxThreadGroupCountPerDimension = 65535; // GPU每维最大线程组数
constexpr uint32 NIAGARA_MAX_GPU_SPAWN_INFOS = 8;             // GPU最多同时处理8批Spawn
```

这些是 GPU 模拟的硬限制，影响 Compute Shader 的 dispatch 参数。

```cpp
constexpr ETickingGroup NiagaraFirstTickGroup = TG_PrePhysics;
constexpr ETickingGroup NiagaraLastTickGroup = TG_LastDemotable;
```

Niagara 在 UE 的 Tick 顺序中占据**从 PrePhysics 到 LastDemotable** 的范围，这意味着它可以在物理之前或之后更新。

## 二、核心枚举（你会反复碰到的）

### `ENiagaraSimTarget`（第 94 行）— 模拟在哪跑

```cpp
CPUSim,          // CPU 模拟（VM 执行）
GPUComputeSim    // GPU 模拟（Compute Shader）
```

这是 Emitter 级别的设置，决定粒子数据走 VM 还是 GPU。

### `ENiagaraScriptUsage`（第 715 行）— 脚本用途，**最重要的枚举之一**

```
Function              → 被其他模块调用的函数
Module                → 可复用的模块
DynamicInput          → 动态输入（节点图中的内联逻辑）
ParticleSpawnScript   → 粒子生成时调用
ParticleUpdateScript  → 每帧更新粒子
ParticleEventScript   → 响应事件
ParticleSimulationStageScript → 自定义模拟阶段
EmitterSpawnScript    → Emitter 生成时调用一次
EmitterUpdateScript   → 每帧更新 Emitter
SystemSpawnScript     → System 生成时调用一次
SystemUpdateScript    → 每帧更新 System
```

对应了 Niagara 编辑器里的**三级结构**：

```
System (Spawn/Update)
  └── Emitter (Spawn/Update)
        └── Particle (Spawn/Update/Event/SimStage)
```

### `ENiagaraScriptGroup`（第 795 行）— 简化版分组

```
Particle = 0, Emitter, System
```

### `ENiagaraDataSetType`（第 130 行）— 数据集类型

```
ParticleData  → 粒子属性数据（位置、速度、颜色等）
Shared        → 共享数据
Event         → 事件数据（Emitter 之间通信）
```

## 三、关键结构体

### `FNiagaraDataSetID`（第 173 行）

```cpp
FName Name;              // 数据集名字
ENiagaraDataSetType Type; // 类型（Particle/Event/Shared）
```

这是数据集的**身份标识**。每个 Emitter 的粒子数据、事件通道都通过它来区分。

### `FNiagaraFunctionSignature`（第 269 行）— 函数签名

```cpp
FName Name;                    // 函数名
TArray<FNiagaraVariable> Inputs;  // 输入参数
TArray<FNiagaraVariable> Outputs; // 输出参数
uint32 bMemberFunction : 1;    // 是否是 DataInterface 的成员函数
uint32 bSupportsCPU : 1;       // 是否支持CPU
uint32 bSupportsGPU : 1;       // 是否支持GPU
```

这是 Niagara VM 和 DataInterface 函数的**接口描述**。当你在节点图中连一个 DI 函数节点时，背后就是这个签名。

### `FNiagaraScriptDataInterfaceInfo`（第 453 行）

记录一个脚本中使用的 DataInterface 的信息：指向哪个 DI 实例、名字、类型。

### `FVMExternalFunctionBindingInfo`（第 570 行）

VM 调用外部函数的**绑定信息**。`InputParamLocations` 中 `true` 表示该参数来自寄存器，`false` 表示来自常量。这是 VM 执行时查找外部函数的关键。

### `FNiagaraVariableAttributeBinding`（第 896 行）

渲染器使用的**变量绑定**。它把渲染器需要的输入（如 Sprite 的位置、颜色）绑定到粒子属性或 Emitter/System 参数上。包含三个层级的变量：

- `ParamMapVariable` — 完整命名空间（含 Emitter 名）
- `DataSetVariable` — 数据集中的名字（不含命名空间前缀）
- `RootVariable` — 用户设置的原始值和默认值

### `FNiagaraSystemUpdateContext`（第 622 行）

编辑器专用的**热更新上下文**。当你在编辑器中修改 Niagara 资产时，它负责收集所有正在运行的实例并 Reset/ReInit 它们，让修改立即生效。

## 四、工具函数 `FNiagaraUtilities`（第 1034 行）

提供了一些全局工具：

- `GetUniqueName` — 生成唯一名字
- `SupportsNiagaraRendering` — 判断当前平台是否支持 Niagara 渲染（需要 SM5 或 ES3.1）
- `SupportsGPUParticles` — 是否支持 GPU 粒子（需要 Compute Shader）
- `BufferFormatToPixelFormat` — Niagara 格式转 UE 像素格式
- `PrepareRapidIterationParameters` — 编辑器用，准备快速迭代参数

---

## 总结

`NiagaraCommon.h` 要记住的核心概念：

1. **三级层次**：System → Emitter → Particle，每级有自己的 Spawn/Update 脚本
2. **模拟目标**：CPU（VM）或 GPU（Compute Shader）
3. **数据集**（DataSet）用 Name + Type 标识，是粒子数据的容器
4. **函数签名**（FunctionSignature）描述 VM/DI 可调用的函数接口
5. **变量绑定**（AttributeBinding）连接渲染器输入和粒子数据
