# 如何向 AI 提问 — 一页纸精简版

> 详细解释见 [[Wiki/Syntheses/Methodology/How-to-prompt-ai-chat|详细版]]。
> 本页设计为一页纸，可以直接截图 / 复制到群里。

---

## 核心心智模型

把 AI 当成一个**刚入职、非常聪明但完全不了解你业务的顾问**。
- 他不知道你的项目、技术栈、背景
- 你每句话都在限定他的回答方向
- 他更倾向于"看起来答上了"而不是说"不知道"
- → **背景给足，目标说清，结果要验证**

---

## 好问题的 5 个要素（至少覆盖"目标"和"背景"）

| 要素 | 说明 | 例子 |
|---|---|---|
| **目标** | 你到底想要什么 | "优化到 100ms 以内，可读性可牺牲" |
| **背景** | AI 不知道的前提 | 技术栈、版本、已经试过什么、约束 |
| **输入** | 原始材料 | 真代码 / 真日志，不要转述 |
| **输出** | 想要的形态 | "表格对比"、"先 TL;DR 再展开" |
| **角色** | 锁定风格（可选） | "你是做 code review 的资深工程师" |

---

## 提问前的检查清单

- [ ] 我说清楚**想要什么**了吗？（目标）
- [ ] 给了**必要的背景**吗？（版本、环境、约束、已尝试）
- [ ] 给了**原始材料**而不是转述吗？（代码 / 日志 / 数据）
- [ ] 说了希望的**输出格式**吗？
- [ ] 复杂任务**拆分**了吗？
- [ ] 有没有让它**先想再答**？（"先列出假设再回答"）
- [ ] 我打算把答案**验证**后再用吗？

---

## 8 条实用技巧

1. **给例子** — 一个例子胜过十句形容词
2. **让它先想** — "列出分析步骤和假设，再给结论"
3. **鼓励反问** — "信息不够的话先问我"
4. **拆分任务** — 架构 → schema → API → …，每步验证
5. **允许说不知道** — "不确定的地方明确标注"
6. **迭代 3-8 轮** — 第一次答不好是常态
7. **指出错误要讲原因** — "不对" ✗ / "你假设了 X 但实际是 Y" ✓
8. **长对话失控时重开** — 让它总结 → 新开对话粘过去

---

## 常见反模式（避免）

- ✗ 只丢关键词，把 AI 当搜索引擎
- ✗ 一句话问复杂问题（"帮我搞定这个 bug"）
- ✗ 转述代码 / 日志 而不是直接贴
- ✗ 贴 3000 行无关代码稀释关键信息
- ✗ 不看回答直接说"错了，重写"
- ✗ 不验证就直接用（代码、命令、API、数字）
- ✗ 相信它"联网查了最新资料"（标准 chat 默认没联网）
- ✗ 用"不要 A"而不是"请做 B"

## 一个对照

**差**：
> 这段代码有问题，帮我看看。[贴 500 行]

**好**：
> UE 4.26 项目，下面函数在 iOS 偶发崩溃（Windows 正常），我怀疑是线程问题。
> 请：
> 1. 先指出线程安全隐患
> 2. 给最小改动修复方案
> 3. 信息不够就告诉我还需要什么
> ```cpp
> [相关的 50 行]
> ```

回答质量差一个数量级，只多花 2 分钟。
