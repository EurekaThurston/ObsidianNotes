## 1. LLM 本质:一个高度复杂的词语接龙引擎

### 1.1 一句话本质

你每天用的 ChatGPT、Claude、DeepSeek、豆包、Kimi,本质上都是 [[Wiki/Concepts/AIApps/Llm|LLM]](Large Language Model,大语言模型)。

**它到底在做什么**:

> **根据前面所有字,猜下一个字最可能是什么。**

就这么简单。一切让它"看起来像在思考"的表现,都是**概率接龙本身涌现出的推理表象**。

### 1.2 ⚠️ 重要澄清:它不"知道"

LLM **没有"真的在想"**,也没有"知道 vs 不知道"的自觉。这是理解它为什么会幻觉、为什么上下文塞满变笨、为什么同一问题两次答不一样的**钥匙**。

- 它不会"卡住"思考——它总能"流畅地"接下去
- 它分不清"我在输出确定事实"和"我在接龙编一个听起来合理的说法"——**这两者对它都是一样的 token 生成**
- 它没有元认知,没有"这个问题我答不好"的自省

**不接受这一点**,你就会被它的笃定语气骗到。

### 1.3 底层架构:Transformer

- **论文**:2017,Google《Attention is All You Need》
- **关键创新**:自注意力机制(Self-Attention),让模型可以"一眼扫过整段话",而不是从左到右逐字
- **变体主导当今所有旗舰**:GPT、Claude、Gemini、Llama、DeepSeek、Qwen、GLM 都是 Transformer 及其变体

### 1.4 处理单元:Token

LLM 不按字母或汉字处理文本,而是按 **token**(子词单元):

- **英文**:1 token ≈ 0.75 单词(常见词)或 1 词(生僻词)
- **中文**:1 个汉字 ≈ 1-2 token
- **代码**:常见变量名、关键字通常是 1 个 token

**为什么要关心 token**:

- **API 按 token 计费**(输入和输出都收费,输出通常更贵)
- **上下文窗口按 token 计量**(见 §2.2)
- **prompt caching 省钱**:相同前缀多次调用可缓存,**这是 Agent Skills 渐进式披露省钱的真金白银**

### 1.5 三步训练:一个 LLM 怎么造出来的

```
Step 1: Pre-training(预训练)
  ├─ 几万亿 token 的互联网语料
  ├─ 任务:"猜下一个字"
  ├─ 花掉成本 90%+
  └─ 造就"通用语感",但此时只会续写,不听指令

Step 2: SFT(Supervised Fine-Tuning,指令微调)
  ├─ 人工写"问题—好回答"样本
  └─ 教它学会对话模式

Step 3: RLHF(Reinforcement Learning from Human Feedback)
  ├─ 人类对 AI 多个候选回答打分
  ├─ 用打分调教模型
  └─ 决定模型的"性格"
```

**这三步解释了很多现象**:

| 现象 | 成因 |
|---|---|
| 各家 AI 性格差那么多 | RLHF 偏好设置不同 |
| 知识有"截止日期" | 预训练语料有时间点 |
| 有时礼貌但答非所问 | RLHF 把"讨人喜欢"训练得胜过"答对" |
| 某些问题拒答 | RLHF 被加了安全约束 |

### 1.6 上下文窗口:一次能看多少

[[Wiki/Concepts/AIApps/Context-window|Context Window]] 是 LLM 一次能同时读取和关注的 token 数量上限。**类比人的短期记忆**——你能记住一页纸,记不住一整本书。

- 早期模型:几千 token(一篇短文)
- 2024 年起:数十万 token(一本书)
- 旗舰模型:百万级 token(一整个代码库)

看起来"变大就变强",但有个反直觉陷阱,见 §2.2。

### 1.7 代表产品(2026)

- **闭源旗舰**:Claude(Anthropic)、GPT(OpenAI)、Gemini(Google)
- **开源主力**:Llama(Meta)、DeepSeek、Qwen(通义千问)、GLM(智谱)
- **国内消费端**:豆包、Kimi、文心一言

### 1.8 小结

- LLM 本质 = 概率接龙,没有"知道"
- 三步训练决定能力与性格
- Token 是计量与计费的基本单位
- Context Window 是视力范围

有了这层基础,我们就能理解为什么它有**结构性怪癖**——下一节。

## 2. 使用 AI 的三个必知怪癖

如果只带走三件事,就是这三个。**每一个都不是 bug,都是结构性问题**,下一代模型只能缓解,不能消除。

### 2.1 幻觉(Hallucination)

#### 2.1.1 定义

[[Wiki/Concepts/AIApps/Hallucination|幻觉]] = **AI 一本正经地胡说**。

因为它的目标函数是"预测下一个 token 的概率",不是"说真话"。当训练数据里没有准确答案时,它会**顺着语气编一个**——而且编得又**流畅又自信**。

#### 2.1.2 典型表现

- 编造不存在的论文、书名、作者、人物
- 编造不存在的 API 函数、库名、命令行参数
- 把 A 的观点安到 B 头上
- 给出错误的日期、数字、统计,但语气确信
- 在解释代码时编造源码里不存在的"设计理由"

#### 2.1.3 为什么是结构性的

- 训练语料本身混杂真假,模型学到的是"看起来合理的表达"
- RLHF 会让模型更倾向"讨好"用户(听起来像个好回答)而非"承认我不知道"
- 只要核心机制还是概率生成,就有幻觉——**"让模型说不知道" 只能缓解,不能消除**

#### 2.1.4 对抗幻觉的四条机制(后面各章展开)

1. **[[Wiki/Concepts/Methodology/Rag|RAG]]**(开卷考试):把可信资料塞进上下文,AI 有东西可抄,胡编空间小
2. **工具调用 / MCP**:让 AI 去真实系统查数据,而不是靠记忆
3. **Harness 反馈回路**:AI 产出被测试/Linter/类型检查自动验证,错了强制改
4. **HITL(Human-in-the-Loop)**:关键决策点叫人审核

推理模型**对事实类幻觉改善有限**——它会更"自洽",不一定更"真"。

#### 2.1.5 ⚠️ 用户第一纪律

> **AI 给出的任何具体事实(数字、日期、链接、引用、API 名、函数签名、人名)都要验证一遍再用。**

这是使用 AI 的**底线**,对任何岗位、任何场景都适用:

- 美术引用参考图/艺术家名 → 验证
- 策划引用游戏案例/销量数据 → 验证
- 开发引用 API 文档 → 验证
- 管理者引用行业数据 → 验证

不守这条纪律,早晚要被坑一次。

### 2.2 Context Rot(上下文腐烂)

#### 2.2.1 定义

当上下文塞到 **80%、90% 满**时,AI 表现会明显下滑:

- 忘记早先说过的约定
- 前后矛盾
- 忽略你刚刚塞进去的关键文档
- 回答开始"走神"

另一个等价说法是 **"Lost in the middle"** ——AI 对**最前面和最后面**的内容记得清楚,**中间段落**容易失焦(和人类阅读长文的心理很像)。

#### 2.2.2 工程意义(极大)

**Context 越大越好**是错的直觉。正确做法是:

- **精准地只给当下需要的那一份**,不是堆所有相关文档
- 这就是 [[Wiki/Concepts/AIApps/Agent-skills|Agent Skills]] 采用 **渐进式披露** 的根本动机(§7)
- 也是 [[Wiki/Concepts/AIApps/Harness-engineering|Harness Engineering]] 四大支柱之一"上下文管理"的核心问题(§6)
- 也是 [[Wiki/Concepts/AIApps/Ai-agent|Agent]] 采用 **subagent 分工** 的动机——让每个子 Agent 只看自己那一小摊(§4)

#### 2.2.3 实际案例

2025 年业界开始注意到:**宣称 1M 上下文的模型,实际有效利用可能远小于此**。业界有 Dex Horthy 提出的 "Smart Zone / Dumb Zone" 经验划分,但具体规律还没有统一结论。

### 2.3 输出不稳定

**同一 prompt 两次问,可能答得不完全一样**——这是由采样温度(temperature)带来的刻意随机性,让生成不至于死板重复。

**对使用者的意义**:

- **重要决策不能靠"一次问了就信"**,多问几次/换问法交叉验证
- 代码、法律、财务等需要**确定性**的场景,要把采样参数调到最保守(或用辅助机制锁定)
- 团队协作时,**同一问题每个人得到的答案可能不同**——这是 wiki(见 [[Wiki/Syntheses/Methodology/Llm-wiki-方法论-读本|方法论读本]])的价值之一:沉淀确定答案

### 2.4 小结:三个怪癖

| 怪癖 | 根源 | 对抗手段 |
|---|---|---|
| 幻觉 | 概率生成 + 语料混杂 + RLHF 讨好 | RAG / 工具 / Harness / HITL |
| Context Rot | Attention 在长序列上失焦 | 渐进披露 / Subagent / 精准注入 |
| 输出不稳定 | Temperature 采样刻意随机 | 多次交叉 / 保守参数 / 沉淀到 wiki |

**第一纪律反复强调**:具体事实必核验。

## 3. 2024-2026 的三个重大变化

### 3.1 推理模型(Reasoning Model)

#### 3.1.1 定义

会"**先打草稿再答题**"的新一代 LLM。相比过去模型看到问题立刻开始写,[[Wiki/Concepts/AIApps/Reasoning-model|推理模型]] 会先在心里列方案、试错、自我质疑、自我纠正,然后才输出最终答案。

#### 3.1.2 时间线

- **2024 年 9 月**:OpenAI 发布 **o1**,打开类别序幕
- **2025 年初**:**DeepSeek-R1** 开源,性能对标 o1,全球震动
- **随后**:Claude **Extended Thinking** / Gemini **Thinking** / 多家跟进
- **2025 年下半年起**:几乎所有重要模型都支持某种形式的"思考模式"

#### 3.1.3 与过去模型的区别

| 维度 | 过去的 LLM | 推理模型 |
|---|---|---|
| 响应方式 | 看到问题立刻开始写 | 先"思考"再写 |
| 速度 | 快 | 慢很多 |
| 成本 | 便宜 | 贵(思考也烧 token) |
| 数学/逻辑题 | 中等 | 显著提升 |
| 复杂编程 | 中等 | 显著提升 |
| 简单问答 | 不差 | 没优势,反而慢贵 |

#### 3.1.4 机制(简化)

推理模型的"思考"**不是魔法**——本质仍是用 token 写草稿,只不过那部分草稿通常**被折叠起来**不直接展示给用户(有些厂商允许看)。

实现上业界混用两种路线:

1. **显式思维链训练**:RLHF 阶段引入大量带推理过程的样本,奖励"想得清楚"的过程
2. **强化学习搜索**:让模型生成多条推理路径,根据结果反馈强化"好路径"

对用户:**它不是新底模,而是同一底模的不同用法——"给它更多 token 去自己想"**。

#### 3.1.5 用在哪、不用在哪

**用**:数学题、复杂逻辑、多步推演、难编程题、复杂分析
**不用**:简单问答、创意写作、闲聊(慢贵无优势)

#### 3.1.6 与 Agent 的搭配

2025 年后常见:Agent 里做**"规划"**步骤用推理模型,做**"执行"**步骤用普通模型。

#### 3.1.7 ⚠️ 一个陷阱

推理模型对"**编造事实**"类幻觉**改善有限**——它会更"自洽",不一定更"真"。推理过程本身也可能不可靠(草稿里想对了最后写错)。

### 3.2 多模态(Multimodal)

到 2026 年,**多模态已是旗舰模型标配**——可以同时处理文字、图像、音频、视频。

对美术/设计岗位意义重大:

- **图 → 描述 / 评分 / 修改建议**:把一张 concept art 丢给 AI,让它分析构图、配色、风格
- **描述 → 图 / 视频**:text-to-image / text-to-video(包括 Diffusion 系列,见 [[Wiki/Syntheses/AIArt/Lora-深度指南-读本|AIArt 读本]])
- **音频 → 文字**:会议转写、视频字幕

本读本不深入展开多模态技术细节(属于 AIArt / AI 视觉等主题),但需知道**它已经是 2026 年的 baseline 能力**。

### 3.3 从 AI Chat 到 AI Agent 的能力跃迁

这是**2025-2026 年最大的范式变化**,本读本 §4 专门讲。先给概念:

| 阶段 | 时间 | 特征 |
|---|---|---|
| AI Chat(聊天) | 2022-2023 | AI 只给建议,人工复制粘贴执行 |
| AI Composer(编排) | 2024 | AI 能直接改代码/文档,每步需人确认 |
| **AI Agent(智能体)** | **2025-2026** | AI 自动读、改、测试、提交,人只做最终审核 |

**纯聊天**:"帮我查明天天气" → AI 说"我没有联网功能"
**Agent**:"帮我查明天天气,下雨就提醒我带伞并写进日历" → 它自己调天气 API、判断下雨、调日历 API 创建事件、告诉你搞定

这是质变。下一节展开。

## 4. AI Agent:从"会说话的大脑"到"能干活的实习生"

### 4.1 定义

**给 LLM "装上手脚"**——让它不只能说话,还能**做事**。核心是一个循环:**思考 → 行动 → 观察结果 → 再思考**,直到任务完成。

纯 LLM 只是个"会说话的大脑"。[[Wiki/Concepts/AIApps/Ai-agent|Agent]] 把它和工具(文件、API、浏览器、命令行……)连起来,让它可以在环境中**循环地执行多步骤任务**。

### 4.2 Agent Loop(核心循环)

```
┌────────────────────────────┐
│ 1. 接收任务                 │
│ 2. 思考推理、制定计划        │  ← 可能用推理模型
│    (Reasoning Model)        │
│ 3. 调用工具                 │  ← 搜索 / 读写文件 / 调 API / 执行命令
│ 4. 观察结果                 │
│ 5. 判断完成与否              │
│    - 没完成 → 回到 2          │
│    - 完成 → 输出              │
└────────────────────────────┘
```

看似简单,实际意味着 AI 从"客服"进化成了"能完成项目的实习生"。

### 4.3 Subagent(多 Agent 协作)

2025 年起主流做法:**主 Agent 拆任务、派给多个 Subagent 各自专攻**,最后主 Agent 汇总。

**好处**:

- 每个子 Agent 只管自己那一小摊 → **上下文窗口不会被撑爆**(对抗 Context Rot)
- 可以**并行**执行,速度更快
- 失败可以**局部重试**,不影响全局

**典型实现**:Claude Code 的 subagent 机制、CrewAI、AutoGen。

### 4.4 Agent 周边概念的定位

| 概念 | 是什么 | 比喻 |
|---|---|---|
| LLM | 只会说话的大脑 | CPU |
| **Agent** | 装了手脚、会循环执行的 LLM | **能干活的实习生** |
| Harness | 围绕 Agent 的约束和基础设施 | **操作系统** |
| Skill | 专业知识包 | **实习生手上的操作手册** |
| MCP | Agent 调工具的标准接口 | **USB-C** |

### 4.5 小结

- Agent = LLM + 工具 + 循环
- Subagent 分工对抗 Context Rot 并支持并行
- 后面的 MCP / Skills / Harness 都是为了让 Agent 跑得**又稳又好**

## 5. AI 连接外部世界:三把钥匙

Agent 要干活,必须能读外部数据、调外部工具、记住上下文。业界演化出三条主要机制:

### 5.1 Function Calling → MCP(让 AI 主动去取)

#### 5.1.1 Function Calling(2023 年)

OpenAI 推出,让 LLM 可以**调外部工具**。开发者注册一组函数描述,模型在对话中生成"要调哪个函数、参数是什么"的结构化输出,应用层执行后把结果回塞给模型。

**问题**:**每家 AI 公司的接口格式都不一样**——给 GPT 写的工具用不了 Claude,反过来也一样。**N 个 AI × M 个工具 = N × M 套定制集成**,成本爆炸。

#### 5.1.2 MCP(2024 年 11 月)

[[Wiki/Concepts/AIApps/Mcp|MCP]](Model Context Protocol)是 **Anthropic 推出的开放标准协议**,专门解决 N × M 难题。

**类比**:

> **AI 世界的 USB-C。**

USB-C 之前:鼠标 PS/2、打印机并口、手机各种充电线。USB-C 统一了所有。**MCP 对 AI 做的是同样的事**。

#### 5.1.3 三个角色

- **MCP Host(宿主)**:发起请求的 AI 应用(Claude Desktop、Cursor、IDE 等)
- **MCP Client(客户端)**:宿主内部与 Server 保持连接的部分
- **MCP Server(服务器)**:对外提供工具和数据的那一端(文件系统、GitHub、Figma、数据库、自定义业务系统……)

**好处**:

- 工具商**只需写一次 MCP Server**,所有支持 MCP 的 AI 都能用
- AI 应用**只需实现一次 MCP Client**,就能挂上全世界的 MCP Server

#### 5.1.4 行业采纳

- OpenAI、Google DeepMind、Cursor、GitHub、Vercel 等陆续支持
- **2025 年后事实上已是行业标准**
- 具体治理(是否捐赠给某基金会等组织细节)以 Anthropic 官方公告为准

### 5.2 RAG(把数据塞给 AI)

#### 5.2.1 定义

[[Wiki/Concepts/Methodology/Rag|RAG]](Retrieval-Augmented Generation,检索增强生成):把文档切块、建索引,查询时检索相关片段注入 prompt,由 LLM 生成答案。

**比喻**:**开卷考试**——模型脑子里可能没答案,但把书摊在它面前它就能查。

#### 5.2.2 它到底怎么"检索"的:Embedding

RAG 检索的底层机制是 **Embedding(向量嵌入)**:

- 把每段文字转成一串数字(一个几百到几千维的向量)
- **意思相近的文字在数字空间里距离也近**
- 检索 = 在向量空间里找"离问题最近的几段文字"

找到的是**"语义相关"**内容,不是关键词匹配——这是 RAG 比传统搜索强的核心。

#### 5.2.3 主要用途

- 对抗幻觉(开卷可抄)
- 把企业内部文档接入 LLM
- NotebookLM、ChatGPT 文件上传、大多数企业知识库都用这套

#### 5.2.4 局限

- **查询时现找**,无长期积累(RAG 的根本限制)
- 对此问题的另一种解法:**LLM Wiki 方法论**(见 [[Wiki/Syntheses/Methodology/Llm-wiki-方法论-读本|方法论读本]])
- 一个更直接的对比表:

| 维度 | RAG | [[Wiki/Concepts/Methodology/Llm-wiki-方法论\|LLM Wiki]] |
|---|---|---|
| 知识积累 | 无 | 持续复利 |
| 交叉引用 | 查询时再找 | ingest 时已建立 |
| 矛盾标注 | 无 | 显式记录 |
| 合成 | 每次重做 | 一次编译,长期更新 |

### 5.3 对话历史 / 长期记忆

短期:上下文窗口保留对话历史(模型自带)。
长期:需要外部机制(向量数据库、文件系统、个人 wiki 等),让跨会话记忆成为可能。

### 5.4 三条路径的对比

| 机制 | 本质 | 类比 |
|---|---|---|
| MCP | Agent **主动去取**数据和工具 | 去图书馆借书 |
| RAG | 把数据**塞**给 AI | 把书放在桌上让它看 |
| 长期记忆 | AI **记得**你上次说过什么 | 认识这个人很久了 |

三者**互补不替代**。一个成熟的 Agent 系统往往三者都用。

## 6. 工程方法论三段论:Prompt → Context → Harness

AI 应用开发的思路,三年走了三个大台阶。这是 **2022-2026 年最关键的主线叙事**,也是本读本跨 Part 的脊椎。

### 6.1 总览

| 阶段 | 时间 | 核心问题 | 类比 |
|---|---|---|---|
| **Prompt Engineering** | 2022-2024 | 怎么问? | 教马"左转""右转"的口令 |
| **Context Engineering** | 2025 | 让 AI 看到什么? | 给马一张地图,让它自己看路 |
| **Harness Engineering** | 2026 | 在什么条件下运行? | 给马装上缰绳、马鞍、护栏、赛道、裁判、计时器 |

### 6.2 第一阶段:Prompt Engineering(2022-2024)

#### 6.2.1 核心问题:怎么问

LLM 刚普及时,人们发现"**同一个问题换个问法,AI 回答质量天差地别**",于是诞生了提示词工程——通过精心设计提示词,让模型输出更好的结果。

#### 6.2.2 常见技巧

- **角色设定**:"你是一位资深 UE4 技术美术"
- **思维链(Chain of Thought, CoT)**:让它"一步步思考"(Wei et al., Google Brain, 2022)
- **少样本示例(Few-shot)**:给几个输入-输出例子当示范
- **格式指定**:要求输出 JSON / Markdown 等结构化格式

#### 6.2.3 局限

不管你多会措辞,AI **仍然看不到你的代码库、业务文档、历史决策**。"一次问一次答"的模式有天花板。

#### 6.2.4 一条有趣的演进

CoT 后来被"**内化到模型训练里**",催生了专门的[[Wiki/Concepts/AIApps/Reasoning-model|推理模型]]——**这是提示技巧升级为架构设计的典型例子**。"CoT prompting" 从"用户要写"变成"模型自己会"。

### 6.3 第二阶段:Context Engineering(2025)

#### 6.3.1 核心问题:让 AI 看到什么

2025 年,[[Wiki/Entities/Methodology/Karpathy|Andrej Karpathy]] 提出:

> **Context Engineering > Prompt Engineering**

Shopify CEO Tobi Lütke 也大力推广。关注点从"怎么措辞"转向"**如何给 AI 搭建一个完整的信息环境**":

- 用 RAG 把相关文档塞进去
- 用 MCP 接通外部工具和数据
- 管理对话历史、长期记忆
- 精心设计系统提示词(System Prompt)

#### 6.3.2 局限

交互模式还没变——**依然是"人给信息,AI 生成内容"**。一旦 AI 要自主跑多步任务(即 Agent 时代),光有好上下文**不够**,还得有约束、有反馈、有护栏。

### 6.4 第三阶段:Harness Engineering(2026)

#### 6.4.1 核心问题:在什么条件下运行

2026 年 2 月,这个词从一篇博客出发,几周内变成整个 AI 工程圈的**高频术语**。

#### 6.4.2 起源

- **Mitchell Hashimoto**(HashiCorp 联合创始人、Terraform 作者)2026-02 博客《My AI Adoption Journey》首次命名并定义,关键原则:

  > **每当发现 Agent 犯错,就花时间设计一个机制,确保它永远不会再犯同样的错。**

- 几天后 **OpenAI** 发布报告:3 名工程师 5 个月不写一行代码,纯靠 Agent(Codex)生成约 100 万行代码,交付真实产品内测版
- **Martin Fowler** 深度分析,将 Harness 归纳为三大类;叠加 OpenAI 强调的反馈回路,形成"**四大支柱**"(见 §7)

#### 6.4.3 LangChain 的有力佐证

**LangChain 的编码 Agent** 在 **Terminal Bench 2.0** 上,底层模型**一个参数都没改**,只优化 Harness(文档结构、验证回路、追踪系统):

- **排名从全球第 30 跃升至第 5**
- **得分从 52.8% 升到 66.5%**
- (具体数字以原博客为准)

这件事的核心洞察:

> **决定 Agent 表现好坏的最大变量,往往不是模型本身多聪明,而是模型被放在什么样的环境里。**

这直接催生了下一节"2026 技术栈三层全景"的认知框架。

### 6.5 三段论的启示

1. **每阶段解决的问题层次不同**:第 1 阶段是"措辞",第 2 阶段是"环境",第 3 阶段是"基础设施"
2. **并非向后取代**——有了 Harness 不代表 Prompt 不用学。新手仍需三阶段基本功
3. **模型层正在大宗商品化**,各家能力越来越接近,**差异化在上面两层**

## 7. Harness 四大支柱(展开)

[[Wiki/Concepts/AIApps/Harness-engineering|Harness Engineering]] 的具体内容就是这四根支柱。每根都在回答:**Agent 会在哪里翻车,怎么用工程手段防止**。

### 7.1 支柱一:上下文管理(Context Management)

#### 7.1.1 问题

Agent 只要**当前任务所需的信息,不多不少**。

把所有规则塞一个大文件里会**失控**:
- 被模型忽略
- 污染其他任务
- 触发 Context Rot

#### 7.1.2 解法:分层上下文 + 渐进式披露

- OpenAI 用 `AGENTS.md` 作为**动态反馈循环文件**
- Anthropic 用 `CLAUDE.md` + **大量分层 README** + 频繁更新的进度文件
- 系统级解法:[[Wiki/Concepts/AIApps/Agent-skills|Agent Skills]] 的**三层加载**(§8)

### 7.2 支柱二:架构约束(Architecture Constraints)

#### 7.2.1 核心原则

把规则"**代码化**"——**用机器而不是人来守边界**。

人类**盯不住** AI 写的大量代码;机器可以。

#### 7.2.2 典型做法

- **Linter 规则**:如"service 层不能反向依赖 controller 层",违规代码合不进来
- **类型系统**:编译期拦截类型错误
- **Schema 校验**:API 请求/响应的结构强制
- **CI/CD 的最后防线**:提交时跑全套自动化检查

#### 7.2.3 OpenAI 的元层次经验

> **让 AI 自己写 Linter 来约束 AI**

这是 Harness 的一个精彩做法——把"犯错→建机制"这个循环**本身也交给 AI 辅助**。

### 7.3 支柱三:反馈回路(Feedback Loops)

#### 7.3.1 核心模式

```
AI 写代码 → 跑测试 → 收到报错
         ↑                    ↓
         └─── 自动修正 ←── AI 读错误
```

让 AI 从"**一次性输出**"进化为"**自我修复的闭环**"。

#### 7.3.2 关键洞察

**失败信息本身就是下一轮最好的上下文**——Linter 的错误消息不只是拦截违规,还在告诉 AI:

- **为什么错了**
- **如何修复**

所以好的 Harness 工具不光拦截,**还给出可读的错误消息**。

### 7.4 支柱四:熵管理 / 垃圾回收(Entropy Management)

#### 7.4.1 问题

AI **长时间运行**会积累"技术债":重复代码、废弃文件、命名不一致、过时注释。

**不定期清理就越跑越烂**。

#### 7.4.2 做法

- **定期 refactor 作业**
- **专门的 cleanup subagent**
- CI 中的 **dead code / unused import 检查**
- **文档与代码的一致性校验**

### 7.5 HITL(Human-in-the-Loop)

#### 7.5.1 定位

Harness 的**常见组件**:在关键决策点**强制让人按确认键**。

典型检查点:
- 要**删数据**
- 要**花钱**
- 要**对外发消息**
- 要**操作生产系统**

#### 7.5.2 理念

**不是倒退回人工**,而是:

> **让人只在真正重要的地方出现。**

这是成熟 Agent 系统的标配。

### 7.6 第五柱的候选:可观测性 / Tracing

2026 年后半段业界开始讨论"**Observability / Tracing 是否应成为第五柱**"。核心需求:

- Agent 执行过程**可追溯**
- 失败时**可诊断**
- 多 Agent 协作**可审计**

尚未形成共识,但势头很强。

## 8. Agent Skills:模块化专业知识

### 8.1 问题

每次和 AI 协作都要重复输入一大堆规范、术语、流程——**效率低且不一致**。

### 8.2 Skill 是什么

[[Wiki/Concepts/AIApps/Agent-skills|Agent Skill]] 的本质:

> **一个带 `SKILL.md` 指令文件的文件夹**,可能还包含模板、脚本、参考资料。

一句话:Skill 不直接解决问题,而是**通过注入合适的知识和流程,把 AI 变成解决这个问题的专家**。

**对美术/设计读者的类比**:Skill 就像 Photoshop 的"**动作面板(Actions)**"或 After Effects 的预设——把一套繁琐流程一键调出来,换个素材照样跑。

### 8.3 核心设计:渐进式披露(Progressive Disclosure)

**这是 Skills 最巧妙的地方**,直接针对 Context Rot 设计。

**类比 Google Maps 导航**——不会一开始就把全程每个细节念出来,而是先给大方向,快到路口才报详细指令。

#### 8.3.1 三层加载

| 层次 | 何时加载 | 内容 | 资源占用 |
|---|---|---|---|
| **第 1 层:元数据** | Agent 启动时 | Skill 名字 + 一句话简介 | 极低 |
| **第 2 层:指令** | AI 判断任务相关时 | `SKILL.md` 完整内容 | 中等 |
| **第 3 层:资源** | AI 执行中按需读 | 脚本、模板、参考文档 | 按需 |

#### 8.3.2 好处

- **你可以挂几十上百个 Skill 也不影响性能**——平时只占极少 token 维护索引,真要用才展开
- 理论上一个 Skill 包可以打包**无限量的知识**
- 配合 **prompt caching** 省真金白银

### 8.4 行业标准化

**2025 年 Anthropic 公布 Agent Skills 规范后**,Claude Code、Cursor、OpenAI Codex、GitHub Copilot、Vercel 等主流 AI 编程工具陆续支持或正在支持这一格式,**正在成为跨工具可移植的开放标准**。

(具体官方资源以 Anthropic 工程博客及 GitHub 规范仓库为准。)

### 8.5 Skills / MCP / CLAUDE.md 的分工

| 维度 | CLAUDE.md | Skills | MCP |
|---|---|---|---|
| 本质 | 项目级规范文件 | 模块化专家知识包 | 工具接口协议 |
| 加载时机 | 启动时**全量**加载 | **按需**加载 | **按需**调用 |
| 解决什么 | 这个项目怎么做事 | 某个领域的专业知识复用 | AI 与外部系统的通信 |
| 可复用性 | 仅限当前项目 | **跨项目、跨工具** | 跨模型、跨应用 |

**简记**:

> **CLAUDE.md 是项目章程;Skills 是专业手册;MCP 是通信协议。**

### 8.6 本仓库就是一个活生生的应用

本仓库的 `CLAUDE.md` §3.4 "主题读本"规则(见 [[Wiki/Syntheses/Methodology/Llm-wiki-方法论-读本|方法论读本]])本质上就是一个**专业化的项目 Skill** ——它告诉 Claudian"本仓库特有的做事方式",通过 Ingest/Query/Lint 三操作把 LLM Wiki 方法论落地。

## 9. OpenClaw:个人本地化 Agent 的代表

### 9.1 要解决的问题

[[Wiki/Entities/AIApps/OpenClaw|OpenClaw]](原名 Clawdbot,因图标是只龙虾被中文社区昵称"**小龙虾**")的出发点:

> **现有 AI 助手都被困在"聊天框"里**——你得在一个对话窗口里跟它说话,它既不能主动干活,也不能操作你的电脑。

### 9.2 定位:AI 助手网关

本质是一个 **AI 助手网关(Gateway)**:

- 一头接 LLM
- 另一头接**各种输入渠道**(网页、飞书、微信、Telegram、Discord……)
- 以及**各种工具**(浏览器、文件系统、命令行等)

### 9.3 核心三件套

**记忆 + 主动 + 行动**,是 OpenClaw 相对普通 AI 聊天的根本差异:

| 能力 | 说明 |
|---|---|
| **记忆**(知道该做什么) | 它记得你上周提过的项目截止日期 |
| **主动性**(知道何时做) | 心跳机制定时唤醒,主动提醒 |
| **行动力**(知道如何做) | 直接操作飞书、邮件、日历完成任务 |

### 9.4 关键对比

| 维度 | 普通 AI 聊天 | OpenClaw |
|---|---|---|
| 运行位置 | 云端 | **你的本地电脑/服务器** |
| 数据隐私 | 发给第三方 | **本地,你完全掌控** |
| 能力范围 | 只能聊 | 操作电脑、浏览网页、读写文件、执行命令 |
| 主动性 | 被动等问 | **心跳机制**定时主动执行 |
| 记忆 | 每次对话独立 | **长期记忆,跨会话保留** |
| 接入渠道 | 单一网页/App | 飞书、微信、Telegram、Discord… |

### 9.5 与飞书的结合

飞书(Lark)承载了大量工作信息。通过飞书官方插件,OpenClaw 可以**以"你的身份"操作**文档、日历、任务。

> **过去需要你手动复制粘贴给 AI 的信息,现在 AI 自己去取。**

### 9.6 ⚠️ 信息缺口

v2 文章从扫盲视角描述,**具体 GitHub 仓库、项目维护者、许可证信息尚未 ingest**。需要时可从官网/社区追溯。

## 10. 2026 技术栈三层全景

这是全文最核心的**架构认知**,也是 LangChain Terminal Bench 实验(§6.4)的自然推论。

### 10.1 三层

```
┌─────────────────────────────────────────────┐
│  第 3 层:Skills(技能层,跨工具可移植)       │
├─────────────────────────────────────────────┤
│  第 2 层:Agent Harness(驾驭层)             │
│  ← 差异化主战场                             │
├─────────────────────────────────────────────┤
│  第 1 层:Model(模型层,正在商品化)          │
└─────────────────────────────────────────────┘
```

### 10.2 每层的现状

| 层 | 例子 | 2026 现状 |
|---|---|---|
| Model | Claude / GPT / Gemini / Llama / DeepSeek | **能力差距快速缩小**,正在商品化 |
| Harness | CLAUDE.md / AGENTS.md / 自家 agent 系统 | **差异化竞争点** |
| Skills | 各公司内部 / 开源 Skill 包 | **正在标准化**(Anthropic 规范落地中) |

### 10.3 战略推论

1. **模型层**:不要 all-in 某一家,各家能力拉平是大势
2. **Harness 层**:这里是**竞争壁垒**——同一底模不同 Harness,体验天差地别(LangChain 实验是铁证)
3. **Skills 层**:**跨工具可移植的专业资产**,长期价值高

**对个人/团队的实践建议**:

- **投入在 Harness 和 Skills** 上,而不是纠结"用哪家 LLM 最强"
- 你的 `CLAUDE.md`、你的 Skill 包、你的 wiki,**这些才是你的专属资产**,LLM 是可替换的后端

## 11. Andrej Karpathy 在 AI 方法论里的位置

**本读本前面多次提到 Karpathy**。他不只是"名人",而是**在 AI 方法论演进里同时留下三个关键节点**的关键人物:

| 时间 | 贡献 |
|---|---|
| 2025 | **Context Engineering 倡导者**(§6.3) — 把焦点从"措辞"转向"完整信息环境" |
| 2025-02 | **Vibe Coding 命名**(§12) — 在推文里首次命名"你主导决策,AI 负责执行"的编程范式 |
| 2026-04 | **LLM Wiki 方法论**(见 [[Wiki/Syntheses/Methodology/Llm-wiki-方法论-读本|方法论读本]]) — 个人知识库的新范式 |

**底层共通判断**:

> **AI 时代人的角色是决策者/策源者,AI 的角色是执行者/维护者。**

三个贡献都是这个判断在不同场景的具体化。这也是为什么本仓库采纳 Karpathy 的方法论——**思路一致**。

## 12. Vibe / Spec / Harness Coding:三种编程范式

这是 AI 编程圈 2025-2026 年的分野。每种范式适合不同场景、不同经验水平。

### 12.1 Vibe Coding(直觉式编程)

- **命名者**:Karpathy,2025-02 推文
- **特征**:**你主导决策,AI 负责执行**。你凭直觉("vibe")描述想要什么,让 AI 实现
- **适合**:快速原型、探索性工作、个人项目
- **风险**:没有规范,复现性差,规模上去会混乱

### 12.2 Spec Coding(规格驱动编程)

- **特征**:**先写清规格(API schema、状态机、业务规则),AI 按规格实现**
- **适合**:团队协作、大型系统、需要长期维护
- **风险**:规格本身写得好坏决定一切,AI 无法补救规格缺陷

### 12.3 Harness Coding(驾驭式编程)

- **特征**:**把 Agent 放在 Harness 里,让 AI 自己完成大部分工作,人在关键节点审核**
- **适合**:已有 Harness 基础设施的团队、规模化 AI 协作
- **风险**:Harness 建设本身是大工程,前期成本高

### 12.4 三者并存

**三种范式并存,不是替代关系**。同一个人/团队可以同时用:

- Prototype 阶段:Vibe
- 设计定型:Spec
- 批量实现:Harness

## 13. 全景回看:一个 AI 应用的完整视角

把前 12 节串起来。以 **Claude Code**(AI 编程 Agent)为例,它是 2026 年一个典型的**集成所有要素**的 AI 应用:

```
┌─────────────────────────────────────────────────────────────┐
│  你在 IDE 里对 Claude Code 说"实现一个用户登录接口"           │
│    │                                                        │
│    ▼                                                        │
│ [Model 层] Claude Sonnet/Opus 底模(可切换)                  │
│    │                                                        │
│    ▼                                                        │
│ [Harness 层]                                                │
│   · 加载 CLAUDE.md(项目章程)                               │
│   · 支柱一(上下文管理):读需要的文件,不读无关的              │
│   · 支柱四(熵管理):保持代码库整洁                           │
│    │                                                        │
│    ▼                                                        │
│ [Skills 层]                                                 │
│   · 按需加载 SKILL.md(如 "TypeScript skill"、"Auth skill")  │
│   · 渐进披露:不用的 Skill 只占索引                          │
│    │                                                        │
│    ▼                                                        │
│ [Agent Loop]                                                │
│   1. 推理模型:规划"要建 route、handler、schema、test"        │
│   2. 调 MCP tool:读文件系统、写代码、改配置                  │
│   3. 观察:跑 test → 失败 → 读错误 → 修(反馈回路,支柱三)    │
│   4. 完成 → 产出 PR                                         │
│    │                                                        │
│    ▼                                                        │
│ [HITL] 你 review PR → merge or 要求改                       │
└─────────────────────────────────────────────────────────────┘
```

**所有 §1-§12 的概念在这一个例子里全都出现了**。

## 14. 八条关键洞察(带走)

1. **LLM 的本质是概率接龙,不是"知道"**——一切困惑都可以回到这一点。它没有元认知,分不清"确定事实"和"流畅编造",**这是幻觉的结构性根源**
2. **三个怪癖是使用 AI 的第一道门槛**:幻觉 / Context Rot / 输出不稳定,都不是 bug,都是结构性问题,**第一纪律:具体事实必核验**
3. **Agent 是 2025-2026 最大的范式跃迁**——从"会说话的大脑"到"能干活的实习生"。LLM + 工具 + 循环 = Agent
4. **三段论演进不是取代,是堆叠**:Prompt → Context → Harness,每阶段解决更高层次的问题,三者都要会
5. **模型层正在商品化,真正的差异在 Harness 和 Skills 两层**——LangChain Terminal Bench 不改模型只优化 Harness,排名第 30→5,这是铁证
6. **Harness 四大支柱**:上下文管理 / 架构约束 / 反馈回路 / 熵管理。Hashimoto 的核心原则:**每次 Agent 犯错,都要建一个机制让它不再犯同样错**
7. **Agent Skills 的渐进式披露**是当前对抗 Context Rot 最成熟的方案——元数据(始终)+ SKILL.md(相关时)+ 资源(执行中),**可挂几十上百个 Skill 也不影响性能**
8. **你的专属资产是 CLAUDE.md + Skills + wiki,LLM 是可替换后端**——这也是为什么本仓库值得存在(见 [[Wiki/Syntheses/Methodology/Llm-wiki-方法论-读本|方法论读本]])
