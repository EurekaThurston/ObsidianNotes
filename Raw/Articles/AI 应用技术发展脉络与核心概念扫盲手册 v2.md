## 第一章：AI 到底在做什么——先把"魔法"拆开

### 1.1 大语言模型（LLM）：一个只会"猜下一个词"的超级大脑

你现在用的 ChatGPT、Claude、DeepSeek、豆包、Kimi，本质上都是同一类东西——**大语言模型（Large Language Model，LLM）**。

它做的事其实只有一件：**根据前面所有的字，猜下一个字最可能是什么**。

就这么朴素。"今天天气真"后面接"好"的概率远高于接"香蕉"，它就写"好"。然后它把"今天天气真好"当作新的前缀，再猜下一个字。一个字一个字滚下来，你就看到它"在写文章"。

之所以感觉像在和一个人聊天，是因为它是读着互联网上几乎所有公开文本长大的——小说、论文、代码、教程、论坛、维基百科。人类在语言里留下的思维模式，它顺着概率都学进去了。

**这一点很重要**：理解 LLM 不是"真的在思考"，而是在做高度复杂的"词语接龙"，你后面很多困惑（它为什么会胡说？它为什么前后矛盾？它为什么忽然变笨？）都会迎刃而解。

代表产品：OpenAI 的 GPT 系列、Anthropic 的 Claude 系列、Google Gemini、Meta Llama、国内的 DeepSeek、Qwen（通义千问）、GLM 等。

### 1.2 Transformer：让这一切成为可能的发动机

2017 年 Google 发表了一篇论文《Attention is All You Need》，提出了 **Transformer 架构**。今天几乎所有主流 AI 模型的底层都是它——不只是文本，连画图的 Sora、Stable Diffusion 的新版、语音模型、蛋白质预测模型都在用。

它到底厉害在哪？一句话：**它让 AI 可以"一眼扫过整段话"，同时理解里面所有词之间的关系**，而不是像过去那样一个字一个字从左到右慢慢看。

打个比方：过去的 AI 像一个只能用手指按住一行字逐字阅读的小学生；Transformer 让 AI 变成了可以一眼扫过整段、立刻抓住重点的成年人。这个能力叫 **自注意力机制（Self-Attention）**。

技术细节不必纠结，但要记住：**所有后面讲到的"更长上下文""更强推理""更准理解"，根源都是 Transformer 给的这点"一眼扫全文"的天赋。**

### 1.3 Token 与上下文窗口：AI 的"视力范围"

- **Token** 是 AI 处理文本的最小单位，可以粗略理解为"一小段字"。英文 1 个 token ≈ 0.75 个单词，中文 1 个汉字通常是 1～2 个 token。
- **上下文窗口（Context Window）** 是 AI 一次能同时"看到"多少 token。就像人的短期记忆——你能记住一页纸的内容，但记不住一整本书。

早期模型只能看几千 token（一篇短文），现在的旗舰模型可以看几十万甚至上百万 token（一整本书甚至一整个代码库）。

**为什么你要关心这个？** 因为：

- 调用 AI API 是**按 token 收费**的，输入输出都要钱。
- 上下文窗口虽然变大，但**塞得太满 AI 反而会变笨**（这叫"上下文腐烂 / Context Rot"，后面会讲）。
- 后面几乎所有"工程方法论"的演进，都是在围绕一件事：**怎么在有限且易污染的上下文窗口里，塞进最有用的信息**。

### 1.4 模型是怎么"练"出来的：三步走

要理解 AI 的行为方式，就得知道它是怎么被造出来的。一个现代 LLM 的诞生大致分三步：

1. **预训练（Pre-training）**：把几万亿 token 的互联网文本喂给它，让它学会"猜下一个字"。这一步花掉整个成本的 90% 以上，造就它的"通用语感"。练完后它能续写任何文本，但**不听指令**——你问"什么是狗"，它可能给你续写成一段关于狗的百科条目。
2. **指令微调（SFT，Supervised Fine-Tuning）**：用人类写的"问题—好回答"对话样本，教它学会"听话"：遇到问题就回答问题，不要乱续写。
3. **人类反馈强化学习（RLHF）**：让人类对 AI 的多个回答打分（"这个回答更有用""这个更礼貌""这个在撒谎"），用打分结果反过来调教模型。这一步决定了模型的"性格"——Claude 为什么显得谨慎，GPT 为什么显得圆滑，很大程度上来自这一步的偏好设置。

**记住这三步**，你就能理解几个高频问题：

- 为什么不同公司的 AI 性格差那么多？—— RLHF 训出来的偏好不一样。
- 为什么 AI 的知识有"截止日期"？—— 预训练时语料就那么多，之后的事它不知道。
- 为什么它有时会"假装"很有礼貌但答非所问？—— 因为它被训练成"讨人喜欢比答对更重要"的倾向是真实存在的。

## 第二章：必须先知道的三个"怪癖"

在学任何具体技术之前，必须先记住 AI 的三个天生毛病。不理解这三条，后面所有工程方法论都会像无根的水。

### 2.1 幻觉（Hallucination）：它会一本正经地胡说

AI **没有"知道"与"不知道"的概念**——它只是在猜下一个词。当它的训练数据里没有准确答案时，它也会顺着语气编一个出来，而且编得**又流畅又自信**。

典型表现：

- 编造不存在的论文、书名、人物、API 函数。
- 把 A 的观点安到 B 头上。
- 给出错误的日期、数字，但语气确信。

**幻觉是 LLM 的结构性问题，不是 bug，不会被"下一代模型"完全消灭。** 所有后面会讲的 RAG、工具调用、反馈回路、人类审核，本质都是在和幻觉作斗争。

**对普通用户的底线**：**AI 给的任何具体事实（数字、链接、引用、代码 API）都要验证一遍再用。** 这是使用 AI 的第一纪律。

### 2.2 上下文窗口虽大，但会"腐烂"

前面讲过 AI 能看很长的内容了。但实际使用中，**把上下文塞到 80%、90% 满，AI 的表现会明显下滑**，表现为：

- 忘记早先说过的约定。
- 前后矛盾。
- 忽略你塞进去的关键文档。

这个现象叫 **Context Rot**，也叫 "Lost in the middle"——AI 对最前面和最后面记得清楚，中间的容易失焦（和人类一样）。

这就是为什么后面的"Skills 渐进式披露""上下文工程"会成为显学：**不是给 AI 越多信息越好，而是要精准地只给当下需要的那一份。**

### 2.3 不稳定：同一个问题，两次答不一样

LLM 生成过程自带随机性（有个参数叫 Temperature 控制这个"想象力"）。同一个问题问两遍，答案经常不一样。对创作场景是优点，对需要稳定输出的场景（比如自动化流程）是问题。

这就是为什么企业级应用里会要求：**输出结构化格式（如 JSON）、加校验、加重试、必要时人工复核**。

## 第三章：AI 的新面孔——推理模型与多模态

2024–2026 年 AI 领域的两个大变化，很多扫盲文章还没跟上，但你应该知道。

### 3.1 推理模型（Reasoning Model）：会"先想一想再答"的 AI

2024 年 9 月，OpenAI 发布了 **o1**。几个月后，DeepSeek 发布了震动全球的 **DeepSeek-R1**（开源且性能对标 o1），Anthropic 也为 Claude 加入了 **Extended Thinking（扩展思考）** 模式。

这一类模型统称 **推理模型**。和过去的 LLM 的区别是：

**过去的 LLM**：看到问题立刻开始写答案。
**推理模型**：看到问题后，先在心里"打草稿"——列方案、试错、自我质疑、自我纠正——然后才给最终答案。

对用户来说体感是：速度慢了、贵了，但**数学题、逻辑题、复杂编程题的正确率大幅提升**。

关键点：推理模型的"思考"并不是魔法，本质还是用 token 写草稿，只不过那部分草稿被折叠起来不直接展示。但它已经从"帮你写文案的工具"进化到"能独立攻克难题的工具"。2025 年之后几乎所有重要模型都支持某种形式的推理模式。

### 3.2 多模态（Multimodal）：它不只能读字

早期 AI 只会处理文字。现在的旗舰模型是**多模态**的——能同时处理文字、图片、音频、视频。

- GPT-4o、Gemini、Claude 都能**看图**：你发一张截图、一张手写便签、一幅画，它能描述、分析、找 bug。
- Sora、可灵、Veo 等模型能**生成视频**。
- ElevenLabs、GPT-4o Voice、豆包语音大模型让 AI **会说话、会听懂口气**。
- Stable Diffusion、Midjourney、Flux 让 AI **画图**。

对美术 / 设计岗位尤其要注意：AI 已经不是"只能写文案的工具"，它已经能看懂你的参考图、给你描边、分析构图、生成分镜、给 UV 贴图做初稿。

## 第四章：从"聊天"到"干活"——AI Agent（智能体）

### 4.1 什么是 Agent

前面讲的 LLM 是"只会说话的大脑"。**Agent 是给这个大脑装上了手脚**——让它不只能说话，还能**做事**。

核心循环：**思考 → 行动 → 观察结果 → 再思考 → 再行动**，直到任务完成。

对比：

- **普通聊天**：你问"帮我查明天天气"，AI 说"我没有联网功能"。
- **Agent**：你说"帮我查明天天气，如果下雨就提醒我带伞，并写进日历"，它会自己调天气 API，看到下雨，调日历 API 创建事件，最后告诉你已经搞定。

### 4.2 Agent 的三代演进

| 时间 | 阶段 | 特征 |
|---|---|---|
| 2022–2023 | **AI Chat（聊天）** | AI 只给建议，你手动复制粘贴执行 |
| 2024 | **AI Composer（编排）** | AI 能直接修改代码/文档，但每一步要你确认 |
| 2025–2026 | **AI Agent（智能体）** | AI 自动读取、生成、调试、提交，人只做最终审核 |

### 4.3 Agent Loop：让 AI 可以处理"多步骤任务"

一个 Agent 的内部大致是这样一个循环：

1. 接收任务
2. 思考推理、制定计划
3. 调用一个工具（搜索、读文件、调 API……）
4. 看工具返回了什么
5. 判断任务完成了没——没完成回到 2；完成就输出结果

这个循环看起来简单，但它意味着 AI 从"一问一答的客服"升级成了"能完成项目的实习生"。后面你会看到的 MCP、Skills、Harness，都是围绕"怎么让这个循环跑得又稳又好"展开的。

### 4.4 多 Agent 协作：主管带实习生

2025 年开始流行的做法是 **多 Agent（或 Subagent）协作**——一个"主 Agent"负责拆解任务、分派活儿，多个"子 Agent"各自专攻一小块，最后主 Agent 汇总。

好处：每个子 Agent 只用操心自己那一小摊事，上下文窗口不会被塞爆；分工后还能并行跑，速度更快。Claude Code 内置的 subagent 机制就是典型代表。

## 第五章：让 AI 连接万物——工具调用、MCP、RAG

Agent 要"干活"，就得能调外部的东西。这一章讲的三个概念，是 AI 和真实世界之间的桥梁。

### 5.1 Function Calling（函数调用）——让 AI 知道"有哪些工具能用"

2023 年 OpenAI 提出的机制：开发者提前给 AI 列一张"工具菜单"（比如：查天气、搜邮件、发消息……），AI 根据用户需求自己判断该调哪个、传什么参数。

问题：**每家 AI 公司的菜单格式都不一样**。给 GPT 写的工具接口，Claude 用不了，反过来也一样。想给所有 AI 都接上同一批工具，就要写 N 遍。

### 5.2 MCP（Model Context Protocol）——AI 世界的 USB-C

2024 年 11 月 Anthropic 推出了 **MCP**，开放标准，用一句话说清楚：

> **MCP 就是 AI 世界的 USB-C 接口。**

USB-C 出现前，鼠标一种线、打印机一种线、手机各种线。USB-C 统一了所有外设。MCP 做的就是同样的事——工具只需要实现一次 MCP 标准，就能被任何支持 MCP 的 AI 应用调用。

三个角色：

- **MCP Host（宿主）**：发起请求的 AI 应用（Claude Desktop、Cursor、各种 IDE）
- **MCP Client（客户端）**：宿主内部与服务器保持连接的部分
- **MCP Server（服务器）**：对外提供工具和数据的那一端（文件系统、GitHub、Figma、数据库……）

影响：OpenAI、Google DeepMind、Cursor、GitHub 等都宣布支持 MCP，2025 年之后事实上已经是行业标准。具体"是否/何时捐赠给某基金会"这种组织细节请以官方公告为准。

### 5.3 RAG（检索增强生成）——给 AI 开卷考试

**问题**：AI 的知识停在训练日期，也不知道你的私有资料（公司文档、个人笔记）。

**做法**：在让 AI 回答前，先从你指定的资料库里**检索**出相关内容，塞进上下文，让它基于这些资料回答。

就像开卷考试——AI 脑子里可能没答案，但你把书摊在它面前它就能查。

**它是怎么"检索"的？** 靠一个叫 **Embedding（向量嵌入）** 的技术——把每段文字转成一串数字（一个几百维的向量），意思相近的文字在数字空间里距离也近。检索就是在向量空间里找"离问题最近的几段文字"。所以 RAG 找到的是**"语义相关"**的内容，不是"关键词匹配"——这是它比传统搜索强的核心。

## 第六章：工程方法论的三次进化（整篇的核心）

AI 应用开发的思路，三年走了三个大台阶。这一节几乎是整篇文章的主心骨。

### 6.1 第一阶段：Prompt Engineering（提示词工程）——"怎么问？"

2022–2024 年，人们发现**同一个问题换个问法，AI 回答质量天差地别**，于是诞生了提示词工程。

常见技巧：

- **角色设定**："你是一位资深 UE4 技术美术"
- **思维链（Chain of Thought，CoT）**：让它"一步步思考"（Google Brain, Wei et al. 2022 提出）
- **少样本示例（Few-shot）**：给几个输入-输出例子当示范
- **格式指定**：要求输出 JSON 或 Markdown

**局限**：不管你多会措辞，AI 看不到你的代码库、你的业务文档、你的历史决策。"一次问一次答"的模式天花板就在那儿。

### 6.2 第二阶段：Context Engineering（上下文工程）——"让 AI 看到什么？"

2025 年，**Andrej Karpathy**（特斯拉前 AI 总监、OpenAI 联合创始人）提出：Context Engineering 比 Prompt Engineering 更关键。Shopify CEO Tobi Lütke 也大力推广。

关注点不再是"怎么措辞"，而是**如何给 AI 搭建一个完整的信息环境**：

- 用 RAG 把相关文档塞进去
- 用 MCP 接通外部工具和数据
- 管理对话历史、长期记忆
- 精心设计系统提示词（System Prompt）

打个比方：
**Prompt Engineering** 是教马"左转""右转"的口令；
**Context Engineering** 是给马一张地图，让它自己看路。

**局限**：交互模式还没变——依然是"人给信息，AI 生成内容"。一旦 AI 要自主跑多步任务，光有好上下文不够，还得有约束、有反馈、有护栏。

### 6.3 第三阶段：Harness Engineering（驾驭工程）——"在什么条件下运行？"

2026 年初，这个词从一篇博客出发，几周内变成全行业高频词。

**起源**：

- **Mitchell Hashimoto**（HashiCorp 联合创始人、Terraform 作者）2026 年 2 月博客首次命名并定义：*"每当发现 Agent 犯错，就花时间设计一个机制，确保它永远不会再犯同样的错。"*
- 几天后 **OpenAI** 发了一篇报告：3 名工程师 5 个月不写一行代码，纯靠 AI Agent（Codex）生成约 100 万行代码，交付了一款真实产品内测版。
- **Martin Fowler** 随后做了深度分析，把 Harness 大致归为三块：**上下文管理、架构约束、垃圾回收（熵管理）**。再叠加 OpenAI 强调的"反馈回路"，业界逐渐形成四大支柱的说法。

**Harness 是什么？**

> **如果 LLM 是 CPU，那 Harness 就是操作系统。**

Harness 是**裹在 AI 周围的整套软件基础设施**——它决定 AI 能看什么、能做什么、怎么拆任务、怎么存记忆、什么时候跑测试、什么时候叫人来审批。

再打个比方：
**Prompt** 是教马口令；
**Context** 是给马地图；
**Harness** 是**给马装上缰绳、马鞍、护栏、赛道、裁判、计时器**——马可以自己全速跑，但跑不出你划的范围。

**四大支柱（综合 Fowler + OpenAI）**：

**支柱一：上下文管理** AI 只要当前任务所需的信息，不多不少。把所有规则塞一个文件里会失控。解法是**分层的上下文 + 渐进式披露**（后面 Skills 那章详讲）。OpenAI 用 `AGENTS.md`，Anthropic 用 `CLAUDE.md` + 大量分层 README。

**支柱二：架构约束** 把规则"代码化"，用机器而不是人来守边界。例子：用 Linter（自动代码检查工具）规定"service 层不能依赖 controller 层"，违规代码根本合不进来。人类盯不住 AI 写的大量代码，但机器可以。

**支柱三：反馈回路** AI 写代码 → 跑测试 → 看报错 → 自己改 → 再跑。失败信息本身就是给 AI 的下一轮上下文。这让 AI 从"一次性输出"进化为"自我修复"。

**支柱四：熵管理 / 垃圾回收** AI 长时间跑会积"技术债"——重复代码、废弃文件、命名混乱。需要定期清理、整理、重构，否则会越跑越烂。

**一个有力的佐证**：LangChain 的编码 Agent 在 Terminal Bench 2.0 上，**底层模型一个参数都没改**，只优化了外围的 Harness（文档结构、验证回路、追踪系统），排名从第 30 跃升至第 5，得分从 52.8% 升到 66.5%（具体数字以原博客为准）。

**这件事证明了 2026 年 AI 工程最重要的一条洞察**：

> **决定 AI 表现好坏的最大变量，往往不是模型本身多聪明，而是 AI 被放在一个什么样的环境里。**

### 6.4 HITL（Human-in-the-Loop）：把人放在关键节点上

Harness 的一个经常出现的组件是 **HITL（Human-in-the-Loop，人在回路中）**——在关键决策点（比如"要删数据库""要花钱""要对外发消息"）强制叫人来按一下确认键。

它不是"倒退回人工"，而是"让人只在真正重要的地方出现"。这是当前所有靠谱 Agent 系统的必备设计。

## 第七章：Agent Skills（智能体技能）

### 7.1 Skills 要解决什么

每次和 AI 协作都要重复输入一大堆规范、术语、流程——效率低且不一致。能不能把这些"专业知识 + 工作流程"**打包成一个可复用的包**，想用就挂上？

**Agent Skills** 就是这个包。本质上一个 Skill 就是一个带 `SKILL.md` 指令文件的文件夹，可能还包含模板、脚本、参考资料。挂上之后，AI 瞬间从"通才"变成"某个领域的专家"。

**一句话**：Skill 不直接解决问题，而是**通过注入合适的知识和流程，把 AI 变成解决这个问题的专家**。

对美术/设计的类比：Skill 就像 **Photoshop 的动作面板（Actions）或 After Effects 的预设**——把一套繁琐的流程一键调出来，换个素材照样跑。

### 7.2 核心设计：渐进式披露（Progressive Disclosure）

这是 Skills 最巧妙的地方。类比 **Google Maps 导航**——它不会一开始就把全程每个细节念出来，而是先给大方向，快到路口才报详细指令。

Skills 分三层加载：

| 层次 | 何时加载 | 内容 | 资源占用 |
|---|---|---|---|
| 第一层：元数据 | Agent 启动时 | Skill 名字和一句话简介 | 极低 |
| 第二层：指令 | AI 判断任务相关时 | `SKILL.md` 完整内容 | 中等 |
| 第三层：资源 | AI 执行中按需读 | 脚本、模板、参考文档 | 按需 |

好处：**你可以挂上几十上百个 Skill 也不影响性能**，平时只留极少 token 的索引，真要用才展开。对照第二章讲的"Context Rot"——渐进式披露就是为了避免上下文被撑爆。

### 7.3 行业标准化

Anthropic 公布 Agent Skills 规范之后，Claude Code、Cursor、OpenAI Codex、GitHub Copilot、Vercel 等主流工具都已支持或正在支持这一格式，正逐步成为跨工具可移植的开放标准。（具体官方资源以 Anthropic 文档为准。）

### 7.4 Skills vs MCP vs CLAUDE.md：别混淆

| 维度 | CLAUDE.md | Skills | MCP |
|---|---|---|---|
| 本质 | 项目级规范文件 | 模块化专家知识包 | 工具接口协议 |
| 加载时机 | 启动时全量加载 | 按需加载 | 按需调用 |
| 解决什么 | 这个项目怎么做事 | 某个领域的专业知识复用 | AI 与外部系统的通信 |
| 可复用性 | 仅限当前项目 | 跨项目、跨工具 | 跨模型、跨应用 |

粗暴记忆：**CLAUDE.md 是"项目章程"；Skills 是"专业手册"；MCP 是"通信协议"。**

## 第八章：OpenClaw（小龙虾）——个人 AI Agent 系统

### 8.1 它要解决什么

现有 AI 助手都被困在"聊天框"里——你得在网页对话窗口跟它说话，它既不能主动干活，也不能操你的电脑。

**OpenClaw**（社区昵称"小龙虾"，因图标是只龙虾）是一个**开源的个人 AI Agent 系统**，部署在你自己的电脑或服务器上运行。通俗说：**一个可以操作你电脑的专属 AI 秘书**。

更本质地讲，它是一个 **AI 网关（Gateway）**——一头接大语言模型，另一头接你的各种入口（网页、飞书、微信、Telegram……）和工具（浏览器、文件系统、命令行）。

### 8.2 和普通 AI 聊天的区别

| 维度 | 普通 AI 聊天 | OpenClaw |
|---|---|---|
| 运行位置 | 云端 | 你的本地电脑/服务器 |
| 数据隐私 | 发给第三方 | 本地，你完全掌控 |
| 能力范围 | 只能聊 | 操作电脑、浏览网页、读写文件、执行命令 |
| 主动性 | 被动等问 | **定时主动执行**（心跳机制） |
| 记忆 | 每次对话独立 | 长期记忆，跨会话保留 |
| 接入渠道 | 单一网页/App | 飞书、微信、Telegram、Discord… |

### 8.3 三个核心差异点：记忆 + 主动 + 行动

- **记忆**（知道该做什么）：它记得你上周提过的项目截止日期。
- **主动性**（知道何时做）：心跳机制定时唤醒，主动提醒。
- **行动力**（知道如何做）：直接操作飞书、邮件、日历完成任务。

### 8.4 和飞书的结合

飞书（Lark）承载了大量工作信息。通过飞书官方插件，OpenClaw 可以以"你的身份"操作文档、日历、任务。**过去需要你手动复制粘贴给 AI 的信息，现在 AI 自己去取。**

## 第九章：AI 编程范式的演进——Vibe / Spec / Harness

（这一章主要面向做软件的人，但其中"分工模式"的演进思路对管理者、美术团队做流程设计也有参考价值。）

### 9.1 Vibe Coding（氛围编程）——2025

**Andrej Karpathy 于 2025 年 2 月在推文里首次命名**（"There's a new kind of coding I call 'vibe coding'"），随后被 Y Combinator 等机构大力推广。

核心理念：**你主导决策，AI 负责执行**。你只说"帮我做一个记账 App"，AI 写全部代码。

很快人们发现问题：降低了门槛，但产出代码质量堪忧——像一个飞快但没工程规范的实习生。

### 9.2 Spec Coding（规格编程）——2025+

先写好详细规格说明（Spec）：架构设计、数据模型、命名规范。AI 按 Spec 执行。质量提升，但整体仍难控制。

### 9.3 Harness Engineering——2026

（第六章已讲）为 AI 构建一整套运行环境：约束、反馈、验证、自愈，让 AI 在护栏内全速跑。

**三阶段演进一句话**：

> **Vibe Coding**："随便跑！" → 乱
> **Spec Coding**："按图纸跑！" → 好一点但还是难管
> **Harness Engineering**："在专业赛道上跑，有护栏、裁判、计时器。" → 可靠、可维护、能规模化

## 第十章：2026 年 AI 技术栈的三层全景

```
┌─────────────────────────────────────────────┐
│  第 3 层：Skills（技能层）                  │
│  可复用的专业知识和工作流                   │
│  编码规范 · 审查标准 · 分析模板 · 部署流程  │
│  跨工具可移植（SKILL.md 开放标准）          │
├─────────────────────────────────────────────┤
│  第 2 层：Agent Harness（驾驭层）           │
│  管理工具、状态、上下文的"操作系统"         │
│  Claude Code · Cursor · Codex CLI · Gemini  │
│  每个 Harness 有自己的编排、记忆、工具管理  │
├─────────────────────────────────────────────┤
│  第 1 层：Model（模型层）                   │
│  底层 LLM 的推理能力                        │
│  Claude · GPT · Gemini · DeepSeek · Llama   │
│  各家差距在缩小——模型正在商品化             │
└─────────────────────────────────────────────┘
```

**核心洞察**：模型层正在"大宗商品化"，各家能力越来越接近。**真正的竞争壁垒在上面两层**——你的 Harness 有多稳、你的 Skills 有多专业。

换句话说：**两个产品可能用同一个底层 LLM，但拥有更好 Harness 和 Skills 的那个，会提供远超对方的用户体验。**

## 第十一章：对不同人意味着什么

### 如果你是美术 / 设计

- AI 已经不只是写文案——它看得懂图、改得了图、生成分镜/参考图/贴图底稿。把它当作一个**永远在线、什么都懂一点的初级助手**，它能接住你不愿手动做的繁琐活。
- **把你的审美标准、流程规范、参考图库打包成 Skill**，给 AI 挂上，它能快速从"通用 AI"变成"懂你项目审美的助手"。这件事 ROI 极高。
- 警惕幻觉：AI 给的颜色参数、尺寸参数、引用的参考图名称都要核实。
- 多模态能力是最值得你亲自体验的一块——拍张手稿发给 Claude 或 Gemini，看看它怎么理解。

### 如果你是开发者

- 角色正在从"代码编写者"转向"**系统设计者**"——设计让 AI 高效可靠工作的环境。
- 学 Harness Engineering 和 Agent Skills 比追新模型更有长期价值。
- MCP 是基础协议，就像 Web 开发者必须懂 HTTP。
- **Subagent 式的分工**值得早学早用。

### 如果你是策划 / 运营 / 非技术角色

- AI 是能帮你干活的"数字同事"，不只是聊天机器人。
- OpenClaw 这类工具可以给你一个 24 小时在线的私人 AI 助手，能处理日历、飞书、邮件。
- **理解这些概念**有助于你在和工程师沟通时讲同一种语言，也有助于你判断哪些需求适合给 AI，哪些不适合。

### 如果你是管理者

- AI 的价值**不在模型多聪明**，而在你如何把它整合进团队工作流。
- **把团队的隐性知识用 Skills 显性化、标准化**，是当前 ROI 最高的 AI 投资方向。
- Harness Engineering 的核心原则——**"人类掌舵，Agent 执行"**——应该是你做 AI 策略的基本框架。
- 别迷信"全自动"。HITL 是成熟 Agent 系统的标配。

## 第十二章：关键概念速查表

| 术语 | 全称 | 一句话 |
|---|---|---|
| LLM | Large Language Model | 海量文本训练的"词语接龙大脑"，能理解和生成语言 |
| Transformer | — | 2017 年提出的模型架构，所有主流 LLM 的发动机 |
| Token | — | AI 处理文本的最小单位，约 0.75 个英文单词或 1–2 个汉字 |
| Context Window | 上下文窗口 | AI 一次能看到的文本长度上限 |
| Pre-training / SFT / RLHF | 预训练 / 指令微调 / 人类反馈强化学习 | 造一个 LLM 的三步 |
| Hallucination | 幻觉 | AI 一本正经胡说的结构性毛病 |
| Context Rot | 上下文腐烂 | 上下文太满 AI 会变笨的现象 |
| Temperature | — | 控制 AI 输出随机性的参数 |
| Reasoning Model | 推理模型 | 先"打草稿"再答题的新一代 LLM（o1、DeepSeek-R1、Extended Thinking） |
| Multimodal | 多模态 | 能同时处理文字、图像、音频、视频 |
| Embedding | 向量嵌入 | 把文本变成数字向量，用于"语义检索" |
| RAG | Retrieval-Augmented Generation | 先检索资料再让 AI 回答，等于开卷考试 |
| Agent | AI Agent / 智能体 | 能自主执行多步骤任务的 AI，不只聊天 |
| Agent Loop | 智能体循环 | 思考 → 行动 → 观察 → 再思考 的核心模式 |
| Subagent | — | 主 Agent 派给子 Agent 分工，避免上下文爆炸 |
| Function Calling | 函数调用 | 让 LLM 调用外部工具/API 的机制 |
| MCP | Model Context Protocol | AI 世界的 USB-C——连接 AI 与外部工具的标准协议 |
| Prompt Engineering | 提示词工程 | 通过优化措辞提升 AI 输出 |
| Context Engineering | 上下文工程 | 为 AI 构建完整信息环境 |
| Harness Engineering | 驾驭工程 | 为 Agent 构建约束、反馈、控制系统 |
| Agent Skills | 智能体技能 | 专业知识/工作流的可复用模块化包 |
| AGENTS.md / CLAUDE.md | — | Agent 运行时参考的项目级规范文件 |
| HITL | Human-in-the-Loop | 在关键节点引入人类审核 |
| OpenClaw | 原 Clawdbot / "小龙虾" | 开源的个人 AI Agent 系统，可本地部署 |
| Vibe Coding | 氛围编程 | 用自然语言描述需求，AI 写代码（Karpathy 命名） |
| Spec Coding | 规格编程 | 先写 Spec，AI 按 Spec 写代码 |
| Linter | 代码检查工具 | 自动检查代码是否符合规范 |
| CI/CD | 持续集成/持续部署 | 代码提交后自动测试和部署的流程 |
