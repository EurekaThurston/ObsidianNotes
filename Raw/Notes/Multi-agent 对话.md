# Multi-agent 对话 — Claudian 解释自己怎么"多 agent"工作

> 2026-04-21,Eureka × Claudian。
> Eureka 问:多 agent 系统到底怎么工作?上下文有限,切成多个 agent 的意义是什么?
> 这份记录是原始对话的要点归纳(非逐字),作为 wiki 的 raw source。

---

## Q1:你(或其他 AI/工具)是怎么做到"多 agent"工作的?规划 agent + 执行 agent 这种链条,上下文不是有限的吗,切多个 agent 的意义是啥?

### 核心反直觉:多 agent 的第一性原理不是"分工并行",是"上下文隔离"

大部分人第一次听"多 agent"会想到多线程并行。但对 LLM 来说,**最珍贵的资源不是算力,是上下文窗口**。而上下文不是越满越好,有三个病:

1. **Context Rot**:窗口越满,对早期 token 注意力越弱、越矛盾、越忘指令。200K 窗口填到 150K,常常比只填 30K 更差。
2. **Prompt Cache 失效成本**:Anthropic prompt cache TTL 5 分钟。上下文结构剧变要重建缓存,直接花钱。
3. **指令漂移**:主任务 system prompt 跟子任务的搜索/浏览噪音混在一起,模型开始"忘了自己是谁"。

**所以"切 agent"的第一性原理是:把脏活、大体量探索、临时性搜索结果,关进一个一次性上下文里,完事只把结论吐回主 agent。**

举例:主 agent 让子 agent 去 codebase 里找所有调用了 X 的地方。子 agent 可能读了 40 个文件、跑 20 次 grep、生成 80K token 中间思考,最后只返回 300 字总结。主 agent 上下文只长了 300 字,不是 80K。

> 主 agent = 项目经理;它不亲自读文件,它派实习生去读,实习生累死自己、把报告交上来就可以被"解雇"(释放上下文)。

### 规划 agent / 执行 agent 链条的真实作用(除了上下文隔离)

1. **角色专用 system prompt + 工具集**:规划 agent 写"你只输出 plan 不写代码";执行 agent 写"你只按 plan 改代码不重新规划"。每个 agent 只拿到自己该有的工具(规划的不给 Write,执行的不给 WebSearch)。
2. **不同模型分工**:规划/审查用贵大模型(Opus),grep/read 苦力用便宜快模型(Haiku)。单 agent 做不到——切模型对话就断。
3. **独立"审视视角"**:审查 agent 看不到主 agent 思考过程,只看结果。这种"没被说服过"的独立视角比同一个 agent 左右互搏可靠——同上下文里的"自我批评"容易被之前推理锚定。
4. **失败隔离**:子 agent 卡住/跑飞/幻觉,主 agent 直接丢掉整个 trace,只记"这条路走不通",自己不被污染。

### 长链条(5 级+)边际收益

通信损耗累积——每层都是"压缩-解压",信息失真。现实常见模式:

- **扁平扇出**:主 agent 并行派 3-5 个子 agent(各自独立上下文),收齐后自己 synthesize
- **两级为主**:planner → N executor → reviewer,很少再嵌
- **循环而非深度**:主 agent 和同一子 agent 反复交互,而不是"子派孙"

Anthropic 自己 multi-agent research 系统的观察:**agent 数量与 token 使用量线性相关,但回答质量非线性提升**,过某点就是浪费。

### 回到问题:"上下文有限,切多 agent 的意义"

**反过来说才对:正因为上下文有限,才必须切多 agent**。

- 单 agent:200K 窗口里塞所有搜索/规划/执行/审查中间产物,很快烂
- 多 agent:每子 agent 自己干净 200K,主 agent 也干净 200K,**总可用"有效上下文" = N × 200K**,代价是 agent 间只能用压缩过的文字通信

这跟操作系统"进程 + IPC"同构:与其搞无限大共享内存,不如每进程自己地址空间 + 明确的消息接口。

## Q2:所以相当于是我给你派活,你自己收到活计划好了之后,又新开几个会话给他们派活对吧?

**完全正确。补几个精确细节:**

### "新开会话"这个词非常准

每个子 agent 本质是**一次全新的 Claude 对话**:
- 不知道用户是谁,不知道之前聊了啥
- 不知道主 agent 的 system prompt(Claudian 的 Obsidian 规则子 agent 也看不到)
- 只能看到**派活那一刻手写给它的 prompt**
- 干完活只返回**一条消息**,然后会话销毁

给子 agent 写 prompt 必须像**给刚走进办公室的新同事交代任务**:背景、目标、已知信息、期望格式,全得写清楚。不能说"你懂的"——它真的啥都不懂。

### 主 agent 看不到工作过程,只看结论

子 agent 可能读 30 文件、跑 50 次 grep、思考 2 分钟——这些全在子 agent 上下文里,主 agent 看不见。子 agent 最后返回的是一段总结文字。

派活时经常加一句"用 200 字以内汇报"——不是省钱,是**防止它把大坨原始信息倾倒回主 agent 上下文**,那就白隔离了。

### 可以同时派多个(并行)

独立任务(如"查 A""查 B"),主 agent 在**同一条消息里**同时发起多个 Agent 调用,并行跑,收齐后综合。有依赖(B 需要 A 的结果)就串行。

### 不是无脑派

派 agent 有成本:通信损耗、prompt 重建、等待时间。只在**值得隔离**时派:

- 要读很多文件 / 大量搜索 → 派(脏活隔离)
- 需要独立审视视角 → 派(避免自我锚定)
- 简单一两步能搞定 → 自己干(派出去反而更慢更容易失真)

### 一个具体例子(Claudian 在本 vault)

> 用户说:"帮我在 vault 里找所有跟'注意力机制'相关的笔记,整理成索引"

Claudian 决策:
- 读很多文件,中间噪音大 → **值得派**
- 派 Explore agent,prompt 写:"在 D:\Notes 搜索所有提到 attention / 注意力 / self-attention 的 .md,按主题归类,返回文件路径+一句话摘要清单,**不要返回原文摘录**"
- 它跑完给主 agent 一张表
- 主 agent 拿这张表跟用户对话、追问、归档

主 agent 上下文只增加了那张表,没被 50 个文件原文淹没。

## 核心隐喻

**"你是包工头,不是工人"**:用户给主 agent 派活,主 agent 拆解后,脏活累活派给一次性临时工干,主 agent 只负责对话、判断、串逻辑。

**操作系统类比**:subagent = 子进程,拥有独立地址空间(上下文),主 agent 通过 IPC(文字消息)与之通信。进程间的"隔离"和"通信受限"都是特性不是 bug。
