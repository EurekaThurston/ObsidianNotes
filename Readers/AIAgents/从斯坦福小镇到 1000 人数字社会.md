## 1. 为什么 2023 年这个 demo 能火

在斯坦福小镇之前,agent-based model (ABM) 已经有几十年历史——NetLogo、Mesa、AnyLogic。几十年学界基本共识:

- agent 的决策规则**手写**:"遇到 X 就做 Y"、"阈值超过 Z 就改变状态"
- 社会学家、经济学家用 ABM 模拟市场、传染、舆论——有用,但**很假**
- 最大痛点:真实人类的决策**千差万别**,规则捕不住

LLM 出来后,一个很直觉的问题冒出来:**能不能用 LLM 替换掉那套硬规则?** 让 agent 的"下一步做什么"直接问 LLM,LLM 看 agent 人设 + 历史上下文自己决定。

但在 Smallville 之前,没人真的把这件事做到**可跨天运行 + 25 个 agent 并行 + 涌现出可信社交行为**的程度。直接把 GPT 套在 agent 身上会撞几个硬墙:

| 硬墙 | 症状 |
|---|---|
| **上下文遗忘** | 第二天 agent 不记得第一天发生了什么 |
| **无长期个性** | agent 行为像"每次重新掷骰子",没有累积人格 |
| **短视** | agent 每 tick 只管当下,没有"今天打算干嘛"的骨架 |
| **记忆污染** | 把全部历史塞进 context → context rot,LLM 变傻 |

这些墙的**每一堵**对应一个架构组件:

- 上下文遗忘 → **Memory Stream**(可持久化 + 可检索)
- 无长期个性 → **Reflection**(从零散记忆提炼人格级洞察)
- 短视 → **Planning**(粗到细递归计划)
- 记忆污染 → **Memory Stream 的三路打分 retrieval**(不是全塞,是按相关性挑)

所以 Smallville 论文的三件套**不是三个可选扩展**——是绕不开的四堵墙的最小化解法。后来大家做类似项目,无非是三件套的不同剪裁,核心没变。

> [!abstract]
> Smallville 2023 真正的贡献不是"跑了 25 个 agent",而是**"长期 agent 运行的最小可行架构"**。把 LLM 从"单轮问答"拓展到"跨天持久模拟"需要什么,这篇论文给了答卷。

## 2. 三件套架构的具体工作原理

### 2.1 Memory Stream — 让"无限历史"装进"有限 prompt"

每个 agent 都有一条**记忆流水**:所有观察、对话、自己做过的事,**以自然语言 + 时间戳追加式**写入。一天下来可能几百条,一周几千条——远远装不进 prompt。

检索时用三路打分:

```
score = w_recency × recency + w_importance × importance + w_relevance × relevance
```

| 维度 | 含义 | 怎么算 |
|---|---|---|
| **recency** | 新的>旧的 | 指数衰减 `λ^(t_now - t_memory)`(λ 典型 0.99) |
| **importance** | 重要事>琐事 | **LLM 自己打分**:入库时喂 LLM "这条记忆 1-10 分有多重要?" |
| **relevance** | 跟当前情境相关的>无关的 | embedding 余弦 |

三分加权求 top-k,注入 prompt。

> [!warning]
> **陷阱 1**:`importance` 用 LLM 打分的成本容易被忽略——每条观察入库都要额外一次 LLM 调用。一天几百条观察,25 agent,就是**每天几千次 importance 打分调用**。
>
> **陷阱 2**:三个权重不是显然的,Park 原版用均权平衡,但不同场景(学校 vs 酒吧)可能需要不同权重。论文附录有扫描,实操时几乎没人调。
>
> **陷阱 3**:embedding 选型影响 relevance 质量。Park 原版用 OpenAI 的 text-embedding-ada-002,换本地模型精度会下降。

**为什么不能简单用 RAG**:标准 RAG 只算 relevance 一维。问题是 agent 模拟里,**最近发生的重要事**(刚刚有人约你吃饭)和**很久以前的无关琐事**(上周吃了什么早饭)如果只比语义,早饭可能因字面接近"吃饭"反而排更高——违背人类直觉。三路打分就是把"时间"和"重要性"显式注入,解决了标准 RAG 的这个偏差。

### 2.2 Reflection — 让"琐碎观察"升级为"人格判断"

原始 memory stream 里条条都是流水账:

```
2023-02-13 08:15  Klaus 在图书馆
2023-02-13 08:30  Klaus 在读 gentrification 论文
2023-02-13 09:00  Klaus 跟同学讨论社区议题
2023-02-13 10:00  Klaus 在咖啡馆
2023-02-13 10:15  Klaus 在笔记本上写论文大纲
...
```

要让 agent 形成**"Klaus 是个认真的社会学研究生"**这种人格级判断,不能 retrieval 硬拼——retrieval 只会返回证据碎片。需要让 LLM **主动从零散证据总结**。

Reflection 触发条件:**最近记忆的 importance 累积超过阈值**(论文值 150)。触发后四步:

1. 扫描最近 100 条记忆
2. LLM 自己生成 **"最值得深究的 3 个问题"**(meta-question)—— 比如"Klaus 对什么感兴趣?"、"Klaus 和谁互动最多?"
3. 针对每个问题在 memory stream 里检索证据
4. LLM 基于证据**生成高阶洞察**,如 "Klaus 在深入研究 gentrification"
5. 洞察本身也是自然语言,**写回 memory stream**,importance 打分(通常较高)

关键洞见:**洞察可以被更高阶 reflection 继续提炼**——reflection 是层级化的。最底层是原始观察,上一层是"今天发生了什么",再上一层是"这周什么主题",最上层可能就是"Klaus 是个什么样的人"。

> [!abstract]
> Reflection 是 memory stream 的**压缩器**——把 100 条"Klaus 在图书馆"压成 1 条"Klaus 认真"。**没有这个压缩,跨天模拟会被 memory 膨胀拖死**——既检索变慢、检索噪声增加,更重要的是 agent 在 retrieval 里看到的永远是细节而不是模式。

Reflection 是整个架构里**最像"意识"的那一层**——它让 agent 具有"自知"(我是谁、我关心什么)。这也是 Smallville demo 里 agent 能说出"我的研究方向是 gentrification"这种话的根源——不是 LLM 随口编,是 reflection 提炼出的东西写回 memory 被 retrieval 命中。

### 2.3 Planning — 让"反应式 agent"具有"长时间尺度"

光有 memory + reflection,agent 仍然是"被动响应":外部来个事件才做决定。但人类一天生活有**内生骨架**——早上起来你就"知道"今天大致要做什么。

Planning 给 agent 这个骨架。递归三层:

```
一天计划(5-10 条,早晨生成)
    │
    ├─ 细化到小时级(每个粗事件展开到多个小时)
    │       │
    │       └─ 细化到分钟级(具体动作 + 位置)
    │
    └─ 执行时事件驱动 reactive 调整
```

触发点:每天早晨 agent 醒来。LLM 看 agent 的 persona + 昨天的 reflections + 明天任何已知约定,生成一天计划。

执行时,另一个 agent 走过来说"下午咖啡馆约一下?",reactive 决定:**重排下午**还是**保持原计划**。这个决定也是 LLM 做。

> [!note]
> 这不是传统的 STRIPS / HTN 规划器——没有显式状态空间、没有目标-子目标分解算法。**全靠 LLM prompt 生成 + 当前 agent 的记忆流上下文。** 所以 "plan" 本身也是一条写入 memory stream 的自然语言记忆,执行时 agent 通过 retrieval 看到自己的计划,保持行动一致性。

这个设计很"偷懒",但 work 得很好——因为 LLM 对"今天要做什么"这种叙事级任务本来就强。代价是 plan 不具备传统规划器的可证明性质(最优、完备),但在 agent 模拟场景不需要最优、只需要合理。

### 2.4 三件套数据流闭环

```
外部事件 / 自己行动
        │
        ▼  新观察(LLM 先 filter 过滤不重要的)
    Memory Stream(三路打分索引 + 持久化)
        │                          ▲
        │ importance 累积过阈值     │ Reflection 输出写回
        ▼                          │
    Reflection ────────────────────┘
        │  层级化洞察
        │
        │ 每天早晨触发
        ▼
    Planning(粗→细)──→ plan 写回 Memory Stream
        │  retrieval 每 tick 读到 plan
        ▼
    Agent 决策:该做什么、说什么、去哪
        │
        └───→ 产生新行动,回到起点
```

这就是"跨天持续运行的 LLM agent"的最小可行闭环。**三件套缺一不可**:

| 少掉 | 后果 |
|---|---|
| Memory Stream | 无历史,agent 每天被"遗忘重启" |
| Memory 的三路打分 | 直接塞全部→context rot,或纯 relevance→时间/重要性信号丢失 |
| Reflection | memory 膨胀拖死 + agent 无人格,只有流水账 |
| Planning | agent 被动反应,没有一天结构,行为随机碎片化 |

## 3. 残酷的经济学:复现为什么是 token 噩梦

代码开源了,能跑,但论文那种 25 agent 2 天完整实验**烧数千美元**——这个数字让很多人复现了个 demo 就放弃。**钱花哪里了?** 这节把成本具体化。

### 3.1 每个 agent 每个 tick 做什么

一个 tick(游戏内几分钟)agent 要跑的 LLM 调用:

```
1. Perceive(过滤):看到周围发生的事情,LLM 判断"这条值不值得存"
   → 1 次 LLM 调用(短 prompt,便宜)

2. Memory 入库 importance 打分:新观察入 memory stream 时
   → 1 次 LLM 调用 / 新观察

3. 决定下一步(最贵的一步):
   - Retrieval 三路打分算好后 top-k 注入 prompt
   - LLM 决定"下一个动作"
   → 1 次 LLM 调用(长 prompt,含人设 + top-k 记忆 + 当前情境)

4. 如果是对话:
   → 每条对话轮次 1 次 LLM 调用(还是长 prompt)

5. Reflection(条件触发):
   - 生成 meta-question(1 调用)+ 每个问题 retrieval + insight 生成(2-3 调用)
   → ~5 次 LLM 调用 / reflection

6. Planning(每天 1 次 + 重排):
   - 一天粗计划(1 调用)+ 小时细化(5-10 调用)+ 分钟细化(20+ 调用)
   → 一天下来 30+ 调用
```

### 3.2 一天成本粗算

假设论文级设定:25 agent × 游戏内 24 小时 × 游戏 tick 每小时约 10 个 → 每 agent 每天 240 tick:

| 环节 | 每 agent / 天 LLM 调用数量 |
|---|---|
| Perceive + importance 打分 | ~500(一天几百条观察) |
| 每 tick 决策 | 240 |
| 对话(估算一天 20 次互动 × 5 轮) | 100 |
| Reflection(一天几次,每次 5 调用) | ~30 |
| Planning(一天一次粗计划 + 多次重排,递归展开 30+ 调用 × 几次重排) | ~50 |
| **合计 / agent / 天** | **~900** |
| **25 agent × 1 天** | **~22,500 调用** |
| **论文 2 天完整实验** | **~45,000 调用** |

每调用 prompt 平均几千 tokens(包含 retrieval 出的记忆 + persona)。按 GPT-3.5 当时定价:

- input $0.0015 / 1K tokens,output $0.002 / 1K tokens
- 平均每次 5K input + 500 output ≈ $0.0085 / 调用
- 45,000 × $0.0085 ≈ **$380**

但这是理想估算。实际:

- Retrieval top-k 扩容、reflection 触发次数多于理论值、重试、失败、长对话 → 实际 token 数更多
- Prompt engineering 迭代期间还要多跑实验调参
- 论文可能跑了远不止 2 天的 replay(附录有更多实验)

**综合:论文级实验 \$1000–\$5000 是普遍估算**。国内复现者"100 秒游戏时间 ≈ 6 元 RMB"是**小规模+本地 LLM**的中间值。

> [!warning]
> 这里是复现的真正劝退点:**不是技术不会,是钱不够**。学生个人项目很难负担。这也解释了为什么后来大家用 llama3 本地跑——不是追求效果,是追求**能跑完**。

### 3.3 哪里能省?

| 优化 | 典型做法 | 代价 |
|---|---|---|
| **本地 LLM** | llama3-70b Ollama / Qwen / DeepSeek | 效果下降(reflection 质量尤其敏感),需要本机 GPU 或 API |
| **小模型分工** | routine 决策用 Haiku 级,reflection / 关键对话用 Opus 级 | 架构改造 + prompt 分工调试 |
| **异步 tick** | 不是每 agent 每 tick 都调 LLM,低重要度时段直接用规则 | 失去"全 LLM 驱动"的优雅 |
| **Memory 压缩** | reflection 后把原始细节删除,只保留高阶洞察 | 失去溯源能力、某些情境下 retrieval 精度下降 |
| **缩规模** | 从 25 agent 缩到 6-10 | 涌现现象可能看不到 |

2026 年社区的主流姿态是**混合**:规模缩(6-10 agent)+ 本地 LLM(llama3)+ 关键点用云(Opus)。[[Wiki/Entities/AIAgents/A16z-ai-town|a16z AI Town]] 的默认配置基本就是这个姿态。

## 4. 2024 跃迁:从"像人"到"特定的人"

2023 Smallville 证明了"LLM agent 能涌现像人的行为"。2024 Park 团队问了一个更硬的问题:**"能不能涌现出像某个特定真人的行为?"**

这两个问题差异比看起来大很多。

### 4.1 评测标准的质变

| 问题 | 2023 | 2024 |
|---|---|---|
| 核心问题 | agent 行为可信吗? | agent 匹配真人吗? |
| 评测方法 | 人类盲测对比 agent vs 真人对话 | agent 答卷 vs 真人答卷对照 |
| ground truth | 模糊(believability 是主观的) | 明确(真人问卷分数) |
| 定量 | 弱(打分 0-5) | 强(GSS 85% 匹配率) |
| 推广 | 需要新的人类评估 | 可按 survey 标准化评测 |

**这个跳变非常关键**。从"主观 believability"到"对照真人 ground truth",意味着:

- 这条技术线**进入严肃社会科学工具箱**,不再只是 HCI 猎奇
- 结果可**复现、可比较、可审查**——学术规范能接上
- 应用场景从"游戏 NPC"扩展到"政策实验、内容推荐、公共健康干预"

### 4.2 方法论转向:Self-Report Grounding

架构上 2024 跟 2023 很不一样——**不再强调 memory/reflection/planning 三件套**,而走 **self-report grounding**:

```
每个真人
  ↓ 2 小时深度访谈(按结构化流程)
  ↓ 完整转录
  ↓
 agent 的 persistent context:
   - 人口学属性(年龄/性别/种族/教育/政治光谱)
   - 完整访谈转录
  ↓
 对新问题 → retrieval 查访谈相关段 + 人口属性 → LLM 生成回答
  ↓
 用真人自己答同一问题的回答作 ground truth 对照
```

关键设计:

- **访谈是 grounding 的本源**——LLM 不是"按人设表演",是"基于访谈回答问题"
- **1052 人样本**按美国人口分层:年龄、种族、性别、民族、教育、政治光谱
- **评测用 General Social Survey**(社会学经典问卷)——**现成标准**,对比空间大

### 4.3 为什么不再是三件套?

服务的问题变了:

- 2023 三件套解决的是"长周期连贯"——**时间维度**上让 agent 过日子
- 2024 self-report 解决的是"个体保真"——**身份维度**上让 agent 像某人

**不需要 agent 跨天演化**,只需要回答问卷时像真人。所以 reflection / planning 都不必要,memory stream 也退化成"访谈转录的 retrieval"。

> [!abstract]
> 这是**架构随问题变化**的好例子。同一作者、同一研究线,架构可以换得很彻底——关键是问题本身变了。读架构论文时容易陷入"三件套是金科玉律"的误区,2024 Park 亲手证明了**架构服从问题,不是反过来**。

### 4.4 85% 匹配率的含义

论文报告 agent 回答 GSS 的**准确率约 85%**,"**跟真人两周后自己再答一遍的 test-retest reliability 相当**"。

这句话两个关键:

1. **85% 不是 100%**——agent 有 15% 的差异,这是 hallucination 的固有量级
2. **真人自己两周后也只答对自己 85%**——因为人类答问卷**本来就不稳定**。所以 agent 的误差**不比真人再答一次更多**

这个对比很聪明——它把 agent 的误差和"真人自己的不稳定"放同一坐标系比较。结果是:**agent 的 15% 差异不比真人本身的噪声更糟**。作为社会科学工具,这已经可用。

> [!warning]
> 当然 85% 不意味着 agent 就能替代真人做任何决策。**敏感议题(健康、司法、政治)上那 15% 偏差可能放大出有害结果**。这是本方向最大的伦理 open question。

## 5. 工业化分支:a16z AI Town 为什么要重写

斯坦福原版代码(`joonspk-research/generative_agents`)是 Python 后端 + Phaser H5 前端。能跑,但**不适合扩散**:

- Python + Phaser 架构松散,研究者用的 Jupyter-ish 风格
- 没有多玩家 / 多机实例支持
- 没有考虑"fork 一下改成自己的小镇"的工作流
- LLM 调用深度绑定 OpenAI,本地模型适配要手动

[[Wiki/Entities/AIAgents/A16z-ai-town|a16z AI Town]] 的作者团队(a16z Yoko Li + Martin Casado + Convex CTO James Cowling)看到的是机会:**把 Smallville 变成一个半小时能 fork 改的 starter kit**。

### 5.1 技术选型的三个关键决定

**决定 1:统一 TypeScript**

前后端一种语言。研究者原版 Python 后端 + JS 前端的二语言栈对非研究者来说是门槛。TS 统一后 Node / PixiJS / LLM 调用 / 数据库 schema 全部同栈。

**决定 2:Convex 作后端**

Convex 是响应式数据库/后端平台,特色:

- **全局共享状态 + 事务**:agent 和玩家同时 commit state,天然支持多 agent + 多人类 input
- **模型天生响应式**:改状态前端自动同步,不用写 WebSocket
- **TS schema-first**:数据结构直接映射 TS 类型

对 AI Town 这种"多 agent + 多玩家 + 实时"的场景**几乎是定制的**。其实 Convex CTO 亲自参与设计,所以这个选型不是意外。

代价:**自建部署要起 Convex 服务或用其云端**——这是锁定,后来被一些分叉版本批评。但对 starter kit 角色来说,降低入门门槛更重要。

**决定 3:默认本地 LLM**

原版默认 GPT-3.5,换本地模型要改代码。a16z 版反过来——**默认 llama3 + mxbai-embed-large 本地跑**,想切云也就改个环境变量。

这个决定跟上一节的"复现太贵"直接挂钩——**作为 starter kit,默认配置必须是本机白嫖能跑的**。不然用户跟着教程跑 10 分钟就烧 5 美元,传播不出去。

### 5.2 和斯坦福原版的架构差异

| 维度 | Stanford 原版 | a16z AI Town |
|---|---|---|
| 架构忠实度 | Memory/Reflection/Planning 全 | **Memory/对话/Plan 保留,Reflection 弱化** |
| Agent 规模 | 25 | 典型 6-10 |
| 目标 | 研究实验 | 可部署游戏/应用 |
| 多人类 | 无(全 NPC) | **支持,人类玩家和 agent 同空间** |
| 持久化 | 简单文件 | Convex 数据库,事务支持 |

**Reflection 弱化是关键妥协**——因为 a16z 面向的场景(几小时玩耍、多人互动)不需要 agent 跨多天演化人格。砍掉最贵的那层,成本压得下来。

> [!abstract]
> a16z AI Town **不是斯坦福小镇的复现**,是**选择性工程化**。砍掉学术场景才需要的部分(25 agent、reflection 层级、长周期运行),保留游戏场景必需的(记忆、对话连贯、计划骨架)+ 加上工业必需的(多人、持久化、易部署)。这是"工程化研究代码"的标准操作——知道砍什么比知道保留什么更重要。

### 5.3 这个选型的**知乎解读文章(p/656007815)**主要讲了什么

根据 anchor 文字 + 同系列文章检索还原,那篇讲的重点:

- 对比 stanford 原版 Python + Phaser vs a16z TS + Convex 的技术栈
- 讲 Convex 作为游戏引擎后端的响应式 / 事务特性
- LLM 接入的抽象层设计
- 前端 PixiJS 地图 / agent 动画的实现
- 如何 fork 改造成自己的小镇

是工程导向的源码解读,和学术意义的讨论不是同一角度。如果你主要想动手改一版自己的小镇,那篇文章有用;如果你想理解三件套架构本身,读 Park 2023 原论文更直接。

## 7. 范式意义:和工具型 multi-agent 的本质区别

本 wiki 的 AIAgents 主题同时容纳**生成式智能体模拟**(本议题)和**项目级 AI 应用落地**(代码问答机器人 / 特效贴图工具)。它们都叫 "agent",但本质不同:

| | 工具型 multi-agent(Claude Code / 代码问答机器人) | 生成式智能体模拟(Smallville / a16z AI Town) |
|---|---|---|
| **核心动因** | 上下文隔离,破解 context rot(见 [[Readers/AIFoundations/为什么上下文有限反而必须切多 Agent]]) | 多 agent 共生演化,观察涌现 |
| **Agent 间关系** | 主从派活,严格层级 | 对等共处,社交网络 |
| **时间跨度** | 单次任务(几分钟到几小时) | 跨天或更长 |
| **通信** | 通过主 agent 中转,避免信息倾倒 | 自然语言直接对话 |
| **评测** | 任务成功率 / token 效率 | 行为可信度 / 真人对照 |
| **架构模板** | 派活 prompt / 角色专用 system prompt / 扁平扇出 | Memory/Reflection/Planning 或 self-report grounding |
| **代表** | Claude Code 子 agent、CrewAI、AutoGen | Stanford Smallville、a16z AI Town、Park 2024 |

同一个词"multi-agent"指两种东西,关键在**第一性原理**:

- 工具型的第一性原理是**上下文隔离**——需要子 agent 是因为 context 有限
- 模拟型的第一性原理是**共生演化**——需要多 agent 是因为要看群体现象

> [!abstract]
> 这个区分在看论文/文章时很有用——看到 "multi-agent" 三个字先看动因。**动因错位导致的架构错位是这方向最常见的讨论混乱**。本 wiki 把两者都放 AIAgents 主题下,是承认它们共享"agent"这个名词,但读者要自觉保留"第一性原理不同"的警觉。

## 10 条关键洞察

1. **三件套不是扩展是必需**:对应 agent 长周期运行的四堵硬墙,缺一就撞墙
2. **Importance 由 LLM 自评**是整个架构的隐藏成本支柱——每条入库都要一次调用
3. **Reflection 是 memory 的压缩器 + 人格生成器**——没它就只有流水账
4. **Planning 不是 STRIPS/HTN**,是 LLM prompt 生成的自然语言计划,写回 memory 由 retrieval 保证执行一致
5. **复现瓶颈在 token 不在代码**——论文级实验数千美元起,所以大家缩规模 / 换本地 LLM
6. **Park 2024 的评测转向(主观 believability → 真人 GSS 对照)比架构变化更重要**——让这方向进入严肃科学
7. **85% 匹配率 ≈ 真人两周后 test-retest**——agent 的误差不比真人自己的噪声更糟
8. **a16z AI Town 是选择性工程化**,不是忠实复现——砍 reflection、砍规模,加多人、加持久化
9. **工具型 multi-agent ≠ 生成式 agent 模拟**:前者第一性原理是上下文隔离,后者是共生演化
10. **2026 实操路径**:先 GenerativeAgentsCN 本机跑、想改读 Park 2023 附录 prompt、想定制场景用 a16z AI Town 作基座
