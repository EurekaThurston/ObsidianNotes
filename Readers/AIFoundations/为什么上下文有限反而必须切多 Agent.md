# 为什么上下文有限反而必须切多 Agent

> 本页是 Multi-agent / Subagent 架构的**主题读本**——详细、精确、满满当当,一次读完即完整掌握"多 agent 到底为什么存在、怎么跑、什么时候派什么时候不派",不需要跳转。
>
> 如需字段级查询或溯源,见末尾的 [[#深入阅读]] 索引。

---

## 0. 本读本要回答的问题

"多 agent" 这个词,第一次听容易被误解成"多线程加速"——就像 CPU 多核同时跑任务。但放到 LLM 语境,这个理解会让你想不通一个反直觉事实:

> [!question]
> 上下文不是有限的吗?切成多个 agent,每个 agent 还是吃自己的上下文,总体不还是那些资源?切了到底图啥?

这其实是一个**完全反着来**的问题——正因为上下文有限,才必须切多 agent。读本要把这个反转讲清楚,并且一路带你答完下面这些关联问题:

- 单 agent 为什么会"越用越笨"?(Context Rot、Prompt Cache TTL、指令漂移三种病)
- 主 agent 派子 agent,子 agent 到底是个什么东西?(字面意义上的"一次全新 Claude 对话")
- 除了上下文隔离,多 agent 还顺带拿到了哪四个收益?(角色 system prompt / 模型分工 / 独立视角 / 失败隔离)
- 为什么 5 级以上的深度嵌套在现实里几乎不存在?(通信损耗累积)
- 什么任务值得派?什么任务派出去反而亏?(决策矩阵)
- 派活的 prompt 要怎么写才不会把信息又倒回主 agent?(派活 prompt 工程)

叙事主线(一张 ASCII 链):

```
单 agent 会得的三种病
      ↓
第一性原理:上下文隔离(不是分工并行)
      ↓
子 agent = 字面意义的新会话,通信必须压缩
      ↓
四个次级收益(顺带拿的白嫖)
      ↓
链路拓扑(为什么扁平 > 深度)
      ↓
派不派的决策 + 实战例子
```

## 1. 单 agent 为什么会"越用越笨":上下文的三种病

要看懂多 agent 为什么存在,得先理解单 agent 在长任务里**为什么不够用**。上下文窗口看起来是个很干脆的数字——"Claude 有 200K token 窗口"——但实际使用时它会患三种病。

### 1.1 Context Rot(上下文腐烂)

窗口越满,模型的表现**不是线性变好**,而是会在某个阈值之后掉头向下。200K 窗口塞到 150K,常常比只填 30K 还更糟。症状:

- 忘记早先说过的约定
- 前后自相矛盾
- 忽略你刚刚塞进去的关键文档(经典的 "lost in the middle")
- 回答开始"走神",风格漂移

> [!warning]
> "我买了 1M 上下文的模型,以后塞啥都不怕"——这是个陷阱。窗口是**物理上限**,不是**有效利用上限**。很多旗舰模型宣称 1M 窗口,实际有效利用可能远小于此。详见 [[Wiki/Concepts/AIFoundations/Context-window]]。

### 1.2 Prompt Cache 5 分钟 TTL

Anthropic 的 prompt cache(以及其他家类似机制)能让"相同前缀重复调用"的成本大幅下降——这是 [[Wiki/Concepts/AIFoundations/Agent-skills|Agent Skills]] 渐进式披露真金白银的价值来源。但缓存有两个硬约束:

- **TTL 只有 5 分钟**:过了就失效,要重新建缓存(花钱)
- **前缀必须严格一致**:上下文结构一动,缓存命中率就崩

这对单 agent 的影响是:一旦你让同一个 agent 既干主任务又干子任务的脏活(搜索、读文件、浏览),上下文结构会**持续抖动**,缓存反复失效,每次重启抖动都要付钱。

### 1.3 指令漂移

主任务的 system prompt 告诉 agent "你是个严谨的代码审查员"。然后你让它先去 grep 一下代码,再去读几个文件,再去浏览一个网页——这些脏活带回来的**噪音**(搜索结果、网页文本、工具输出)不断稀释最前面的 system prompt。

模型会开始**忘记自己是谁**:该保持"代码审查员"人格的时候,可能突然进入"搜索总结员"模式。指令在长上下文里被**稀释成背景噪音**。

### 1.4 三病联合起来看

> [!abstract]
> 单 agent 的"有效上下文"远小于窗口上限。填到 60-80% 就已经在掉性能;而长任务中间会产生**海量一次性垃圾**(搜索结果、读文件、试错思考),这些东西塞进主上下文,三病齐发,agent 迅速变笨。

这就是为什么即使窗口变大,"把所有东西塞给一个 agent"仍然不是好办法。

## 2. 第一性原理:上下文隔离,不是分工并行

理解完上面三种病,多 agent 存在的真正理由才会浮现:

> [!abstract]
> **把脏活、大体量探索、临时性搜索结果,关进一个一次性上下文里,完事只把压缩过的结论吐回主 agent。**
>
> 主 agent 不亲自读文件,它派"一次性实习生"去读。实习生累死自己(读 40 文件、跑 20 次 grep、生成 80K token 中间思考),交 300 字报告后被"解雇"——**主 agent 的上下文只长了 300 字,不是 80K**。

这是 multi-agent 的**第一性原理**,不是"多个 agent 可以并行加速"。并行加速是副产品,不是动机。

### 2.1 有效上下文从 200K 变成 N × 200K

单 agent 里,所有东西(规划 + 搜索结果 + 执行 + 审查中间产物)挤在 200K 里,互相抢空间、互相干扰。

多 agent 里,每个子 agent 有自己的**干净 200K**,主 agent 也保持自己的**干净 200K**。

```
单 agent:  [规划 + 搜索结果 + 执行 + 审查] = 200K 共享抢占
                           ↓
多 agent:  [主规划] = 200K
          [子1搜索] = 200K  ←  三个 "脏" 窗口
          [子2读码] = 200K     任务完成就销毁
          [子3审查] = 200K
```

"有效上下文"从单窗口 200K,放大到 **N × 200K**。

### 2.2 代价:agent 间通信必须压缩

天下没有白占的便宜。多 agent 的代价是:**agent 之间不能共享上下文**,只能通过**压缩过的文字消息**通信。主 agent 不能直接看到子 agent 读过的 40 个文件原文,它只能看到子 agent 给它的那段总结。

这个代价有时候会痛——比如主 agent 事后想追问"你刚才看到的第 27 个文件第 134 行那个注释是什么意思?",它问不了,因为那个子 agent 已经销毁了。这也是本议题还没完全解决的 open question(见 [[#4. 留下的问题]])。

### 2.3 进程 + IPC 的操作系统类比

写过多进程程序的人对这个结构会立刻识别:

```
多进程 OS              Multi-agent
─────────              ─────────
每进程独立地址空间  ←→  每 agent 独立上下文窗口
进程间 IPC(消息)  ←→  agent 间文字消息
fork 子进程        ←→  派子 agent
进程退出释放内存    ←→  子 agent 返回后销毁,释放上下文
```

操作系统不搞"无限大共享内存 + 所有程序挤进去",而是搞"每进程独立地址空间 + 明确的 IPC 接口"——同样的道理,不是因为性能,而是因为**隔离是可维护性的基础**。Multi-agent 走的是一样的路。

## 3. 子 Agent 字面上就是"一次全新的 Claude 对话"

这一节把"子 agent 到底是什么东西"讲到字面程度,顺便回答"派活的 prompt 要怎么写"。

### 3.1 新会话意味着什么

主 agent 发起一个子 agent 调用时,系统实质上是:

1. 开一个**全新的 Claude 会话**(新的 API 调用序列)
2. 用主 agent **手写提供的 prompt** 作为这个新会话的第一条用户消息
3. 配一个这次派活专用的 system prompt(可能比主 agent 的简单得多)
4. 给这个会话**有限的工具集**(比如只给 Read + Grep,不给 Write)
5. 让它自己跑 agent loop(思考→调工具→观察→再思考),直到它决定返回
6. **它返回的消息** = 主 agent 得到的结果
7. 会话销毁,所有中间上下文消失

关键点:**子 agent 不知道**——

- 你是谁(不知道用户)
- 主 agent 在跟你聊什么
- 主 agent 的 system prompt 是什么(Claudian 的 Obsidian 规程子 agent 也看不到)
- 之前的对话历史

**它知道**的只有主 agent 在派活那一刻写给它的那段 prompt。

### 3.2 派活 prompt 要像对刚入职同事

因为子 agent 字面意义上是"陌生人",派活 prompt 必须**自含**。不能说"你懂的"、"按老规矩"、"用我们之前约定的格式",它真的啥都不懂。

典型的派活 prompt 骨架:

```
背景: 这个 vault 是 Eureka 的 Obsidian 仓库,用 CLAUDE.md 规程管理。
目标: 在整个 vault 里找出所有提到 "注意力机制" 的笔记。
已知: 关键词有 attention / 注意力 / self-attention / cross-attention。
      vault 根在 D:\Notes,笔记是 .md 文件。
期望格式: 返回一个表格,列:文件路径 / 一句话摘要。
硬性约束: 不要返回原文摘录,只要摘要。
          总输出控制在 200 行以内。
```

> [!warning]
> 派活 prompt 里"**不要返回原文**""**控制在 N 字/行以内**"这类约束**不是为了省 API 钱**,是为了防止子 agent 把大坨原始信息**倾倒回主 agent 上下文**——那就把"隔离"的意义白白抵消了。
>
> 每次写派活 prompt,都要问自己一句:"这个 agent 拿回来的东西塞进主 agent 的上下文,值不值?"

### 3.3 "会话销毁"意味着 state 不可累积

因为子 agent 跑完就销毁,主 agent **不能**跟同一个子 agent "再聊一轮"——每次派都是**全新的会话**,全新的 prompt,全新的上下文。

这意味着:

- 想让子 agent "基于上次的结果继续做"?**做不到**——必须把"上次的结果"写进新的派活 prompt
- 想让两个子 agent "协作"?**做不到**——它们互相看不见,只能都向主 agent 汇报,由主 agent 综合

这个限制是有成本的(有时候 state 重建很痛苦),但它带来的是**最干净的隔离**——没有 state,就没有 state 污染。

## 4. 除了上下文隔离,还顺带拿到的四个收益

一旦接受了"多 agent"这个架构,四个次级收益会**白白落到口袋里**——它们单独一个都不值得切 agent,但既然切了,不拿白不拿。

### 4.1 角色专用 system prompt + 限定工具集

主 agent 的 system prompt 可能很长(Claudian 这里 200 行的 CLAUDE.md 规程)。但派出去的"代码搜索子 agent"根本不需要知道 CLAUDE.md——它只需要知道"在给定目录里搜索关键词并返回结果"。

于是派活时给这个子 agent 一个**极简的 system prompt**:

```
你是一个代码搜索助手。任务单一:按用户给的关键词在给定目录里搜索,
返回结构化结果。不要主动扩展任务,不要写代码,不要修改文件。
```

再配一个**限定工具集**:

- 规划 agent:只给思考工具,**不给 Write**(物理上不可能改文件)
- 执行 agent:只给 Read / Edit / Write,**不给 WebSearch**(不让它跑去上网走神)
- 审查 agent:只给 Read,**不给任何修改工具**(只读视角)

> [!tip]
> 工具集限制是比 prompt 约束**更硬**的护栏——prompt 说"不要改文件"agent 有时会违规,但**没给 Write 工具**它就是物理上改不了。设计子 agent 时先想工具集,再想 prompt。

### 4.2 不同模型分工

主 agent 可能跑在 Opus(贵 + 聪明),用来做规划、决策、最终综合。但 grep 100 个文件然后返回摘要这种**苦力活**,用 Haiku(便宜 + 快)就够了——甚至可能**比 Opus 干得更好**,因为 Haiku 不会过度思考。

单 agent 做不到这件事——**一旦切模型,对话就断了**(模型之间不共享 KV cache,不共享风格)。多 agent 天然能让不同节点用不同模型。

典型分工:

| 角色 | 模型 | 理由 |
|---|---|---|
| 主 agent(规划/综合) | Opus / Sonnet | 需要深度推理 |
| 搜索/grep 子 agent | Haiku | 只要快,不需要聪明 |
| 审查/lint 子 agent | Opus | 需要独立判断力 |
| 批量改代码的执行 agent | Sonnet | 平衡速度和质量 |

### 4.3 独立"审视视角"(anti-anchoring)

让同一个 agent"自我批评"有个根本毛病:它刚刚做出的推理会**锚定**接下来的批评。"我刚刚论证了这段代码没问题"之后,再问它"这段代码有没有问题?",它倾向于继续论证没问题——即使它本来该发现问题。

独立的审查子 agent **看不到主 agent 的推理过程**,它只看到**结果**。这种"没被说服过"的视角,比左右互搏更可靠。

> [!example]
> Anthropic 官方的 code review 工作流里,审查 agent 只拿到 diff,不拿到写这个 diff 的 agent 的 thought chain——这是故意的,是为了保住独立视角。

### 4.4 失败隔离

子 agent 跑飞、卡死、产生幻觉——主 agent 的反应可以是:**直接丢掉整个 trace**,只记一个简单结论"这条路走不通"。主 agent 自己的上下文**不被污染**。

单 agent 里,一旦跑飞,那坨错误推理就永久留在了上下文里。你想让它"忘掉刚才那段"?非常难——模型会反复被那段错误推理拉回去。

多 agent 的失败是**一次性的**,恢复成本是**零**(重新派一个就行)。

## 5. 链路拓扑:为什么扁平扇出远胜深度嵌套

理论上你可以搞 "主 agent → 子 agent → 孙 agent → 重孙 agent" 5 级嵌套。实际上**几乎不存在**。

### 5.1 通信损耗累积

每一层 agent 间通信都是一次"压缩-解压":子 agent 把 80K 中间思考压缩成 300 字返回,主 agent 解读这 300 字重建理解。单次损耗可以接受,**两次损耗已经明显**,三次以上失真到不可用。

```
L0 主    [完整意图, 100%]
  ↓ 压缩
L1 子    [92% 理解] ──派活──→  L2 孙 [80% 理解]
                                       ↓
                                  L2 孙做事, 返回
                                       ↓ 再压缩
L1 子    [收到: 原意图的 60%]
   ↓
L0 主    [最终收到: 原意图的 45%]
```

每一层都吃**信息税**,层数越多,主 agent 最终拿到的结果越偏离原意图。

### 5.2 三种常见拓扑

实际工程里能看到的结构几乎全是**两级以内**:

**扁平扇出(最常见)**:

```
           主 agent
         /    |    \
       子1   子2   子3     (并行,独立上下文)
         \    |    /
           主 agent 综合
```

**规划-执行-审查(两级线性)**:

```
主 agent → 规划子 agent → 拿到 plan
主 agent → 执行子 agent(执行 plan) → 拿到 diff
主 agent → 审查子 agent(审查 diff) → 拿到 review
主 agent 综合给用户
```

**主与同一子 agent 循环(重复但不深)**:

```
主 agent → 子 agent [v1] → 结果 1
主 agent (用结果1 改派活 prompt) → 子 agent [v2,全新会话] → 结果 2
...
```

注意:**不是"子 agent 再派孙 agent"**,是**主 agent 反复派不同批次的子 agent**。深度永远是 2。

### 5.3 Anthropic 自家观察:agent 数量 vs 质量非线性

Anthropic 2025 年发过一篇讲多 agent research 系统的博文,关键观察之一:

> [!quote]
> Agent 数量和 token 使用量大致**线性相关**,但回答质量**不是**线性提升——过了某个点就是纯浪费。

也就是说,派 10 个子 agent 比派 5 个**花 2 倍的钱**,但质量**提升远不到 2 倍**,甚至可能不提升。

原因很好理解:

- 主 agent 综合 N 个子 agent 报告,N 越大主 agent 自己的上下文越满(回到病 §1.1)
- 更多子 agent 意味着更多通信损耗(§5.1)
- 超过一定数量,子 agent 的工作开始**冗余或冲突**,主 agent 花更多时间调解

所以"尽可能多派"是错的,正确的做法是**精挑细选该派哪几个**。

## 6. 并行 vs 串行:怎么决定

### 6.1 独立任务 → 并行(同一条消息里发起多个 Agent 调用)

主 agent 决定同时派"搜 A 主题"和"搜 B 主题"——这两个任务**互不依赖**,结果可以独立到达。正确做法是在**同一条 API 回复里发起两个 Agent tool 调用**,它们会并行跑,主 agent 同时等两边返回。

这就是 Claudian 的 system prompt 里反复强调的那句:

> [!quote]
> 当你启动多个独立 agent 时,在**一条消息里**发出多个 tool use,它们会并行运行。

### 6.2 依赖任务 → 串行

"先搜出所有相关文件,再对这些文件做 lint"——lint 任务依赖搜索结果,只能串行。主 agent 先派搜索 agent,拿到结果后,再派 lint agent(派活 prompt 里把搜索结果塞进去)。

> [!warning]
> 明明有依赖却并行派,是典型误用——第二个 agent 会缺信息乱跑,浪费 token。
>
> 明明可以并行却串行派,是典型低效——把两段本可重叠的时间串起来,延迟翻倍。

判断依赖的简单办法:**第二个 agent 的派活 prompt 里,是否需要引用第一个 agent 的返回内容?** 需要 → 串行。不需要 → 并行。

## 7. 派不派的决策 + Claudian 的实战例子

### 7.1 决策矩阵

派 agent 是有成本的——通信损耗、prompt 重建、等待时间。**不是越多越好**,而是**值得隔离才派**。

| 情况 | 决策 | 理由 |
|---|---|---|
| 要读很多文件 / 大量搜索 | **派** | 脏活隔离,主 agent 上下文不被淹没 |
| 需要独立审视视角 | **派** | 避免自我锚定,拿到真正的"第二双眼睛" |
| 任务需要专用模型(快 or 便宜) | **派** | 单 agent 无法切模型 |
| 简单一两步能搞定 | **自己干** | 派出去反而更慢更失真 |
| 依赖状态强、需要反复对话 | **自己干** | 子 agent 无状态,反复重建成本高 |
| 需要精确控制格式的小改动 | **自己干** | 压缩通信会丢格式精度 |

### 7.2 Claudian 在本 vault 里的实际例子

> 用户对 Claudian 说:"帮我在 vault 里找所有跟'注意力机制'相关的笔记,整理成一份索引。"

Claudian 的决策链:

1. **判断**:这活要读很多文件,中间会产生大量原文摘录和搜索结果,属于典型脏活 → **值得派**
2. **选 agent 类型**:纯搜索+归类,不需要写代码 → 派 **Explore** agent
3. **写派活 prompt**:

   ```
   背景: 这是 Eureka 的 Obsidian vault(根在 D:\Notes),按 CLAUDE.md 规程组织。
   目标: 找出所有提到"注意力机制"的笔记,整理成主题索引。
   搜索关键词: attention / 注意力 / self-attention / cross-attention / multi-head
   输出格式: 按主题归类的 Markdown 列表
       - 主题 A:
         - [[path/to/file1]] — 一句话摘要
         - [[path/to/file2]] — 一句话摘要
   硬性约束:
     - 不要返回原文摘录
     - 每个文件最多一句话
     - 总输出 200 行以内
   ```

4. **派出 agent 并等待**:Explore agent 可能读 30 文件、跑 10 次 grep、思考几分钟,这些全在**子 agent 自己的上下文**里发生,Claudian 看不见
5. **收到返回**:一张分类表(几百行)
6. **综合**:Claudian 拿这张表跟用户对话、追问、归档,**主 agent 上下文只长了这张表**

对比如果 Claudian 亲自干:Claudian 自己要读 30 文件,每个文件原文都塞进主上下文,读到一半就开始 Context Rot,后续跟用户的对话质量直线下滑。

> [!abstract]
> **包工头不下工地**——主 agent 只负责对话、判断、串逻辑。脏活派出去,拿回结论。这是 multi-agent 架构最核心的心智模型。

## 全景回看

把整张图拼起来,multi-agent 的工作流是:

```
用户问问题
    ↓
主 agent 理解 + 拆任务
    ↓
对每个子任务问:值得派吗?
    ↓
值得 ──┬──► 独立? ──是──► 并行派(一条消息多 tool 调用)
      │                ↓
      │               依赖? ──是──► 串行派(拿到 A 结果再派 B)
      │
      └── 不值得 ──► 主 agent 自己干
    ↓
每个子 agent:
  - 全新 Claude 会话
  - 看到的只有手写派活 prompt
  - 限定工具集
  - 自己跑 agent loop(思考→工具→观察→...)
  - 返回一条压缩消息
  - 会话销毁
    ↓
主 agent 综合所有子 agent 返回
    ↓
回复用户
```

整个过程中,**主 agent 的上下文增长 = 所有子 agent 返回消息总和**(不是所有子 agent 中间思考总和)。这就是"有效上下文 = N × 单窗口"的来源。

---

## N 条关键洞察

1. **多 agent 的第一性原理是上下文隔离,不是分工并行**——并行加速是副产品。
2. **"上下文有限"不是反对多 agent 的理由,而是必须多 agent 的理由**——正因为有限,才要把脏活关进一次性窗口。
3. **子 agent 字面上是一次全新的 Claude 对话**——不知道用户、不知道主 agent 的 system prompt、只看派活 prompt。
4. **派活 prompt 要自含、要带压缩约束**——防子 agent 把原始信息倒回主上下文。
5. **四个白嫖收益**:角色专用 system prompt + 限定工具集、不同模型分工、独立审视视角、失败隔离。
6. **扁平扇出 > 深度嵌套**——通信损耗累积,5 级嵌套几乎不存在。
7. **Anthropic 观察**:agent 数量与 token 线性,但质量非线性——多不等于好,该派则派该省则省。
8. **决策原则**:脏活隔离 / 独立视角 / 专用模型 → 派;状态重、小改动、简单步骤 → 自己干。
9. **OS 类比**:子 agent = 子进程,主 agent 通过 IPC 文字消息通信,隔离和通信受限都是**特性**。
10. **包工头隐喻**:主 agent 不下工地,只负责对话、判断、串逻辑——工地上的脏活派给一次性临时工。

## 自检问题(读完回答)

下面这些题需要把"三种病 / 第一性原理 / 子 agent 本质 / 拓扑损耗 / 派不派决策"几个核心节串起来才能答得有根据。**不要回原文翻答案**——翻得到说明题出得太浅了。

1. **反着设计**:假设 Anthropic 明天发布一个"共享上下文多 agent"架构——所有子 agent 和主 agent 都共享同一份 200K 上下文,看得见彼此的中间思考。这个设计能解决什么问题?又会撞到本读本里哪几节讲的哪几个具体问题?你愿意用这个架构替换掉当前的"隔离 + IPC"架构吗?为什么?

2. **派 vs 不派**:你在给一个复杂 bug 做根因分析,需要同时:(a) 读 15 个相关文件、(b) 跑 git blame 看每段代码的历史、(c) 在一段可疑代码上做 2-3 个小改动试错。三件事里,哪些你会派子 agent、哪些你会自己干?判断标准是什么?(提示:用 §7.1 决策矩阵的几个维度分别对三件事打分)

3. **串起来的连锁后果**:一个用户让 Claudian"整理整个 vault 里所有关于 Niagara 的笔记,写一份总目录"。假设 Claudian 选择**不派子 agent、自己全读**,它的上下文会经历什么序列?把 §1.1(Context Rot)、§1.3(指令漂移)、§3.2(压缩汇报的反向——信息倾倒)三节串起来,描述从"开始读第 1 个文件"到"最终给用户回复"中间会具体发生什么症状。

4. **设计取舍的代价转移**:派活 prompt 里强制"200 行以内压缩汇报"解决了"信息倾倒"的问题。这个约束的代价转移到了哪里?(想想主 agent 事后追问的能力、子 agent 压缩时可能丢掉的信息、信息丢失的不可恢复性) 在什么场景下这个代价会**真的痛**?

5. **拓扑选择**:你要让 AI 帮你做一次大型代码重构,任务分解成"设计新架构 / 写 migration 脚本 / 改动 20 个调用点 / 全量测试 / 写 changelog"5 步。用扁平扇出?用两级 planner-executor-reviewer?用串行循环?还是混合?给出你的拓扑设计,并说明每一步为什么用那种结构。

6. **反直觉核心的五句话解释**:用 **5 句话**向一个完全没写过代码的文科背景朋友解释"为什么上下文有限反而是必须切多 agent 的理由"。要求:不用"上下文窗口""token""context rot"这些黑话,用日常类比讲清楚第一性原理。

7. **四个次级收益的单独必要性**:如果 Anthropic 的 API 突然限制"一次请求只能用一个模型"(§4.2 模型分工失效),又限制"所有 agent 必须用同一个 system prompt"(§4.1 角色专用失效),多 agent 架构还值不值得?哪些收益保留,哪些失效,主 agent 派子 agent 的动机是否还成立?

8. **失败模式**:给一个"多 agent 反而比单 agent 更糟"的具体场景——派了 agent 但效果更差的任务长什么样?至少列两种不同机制下的失败模式,并说明你会怎么在任务一开始就识别出"这个任务不该派"。

## 留下的问题

本议题在 2026 年仍有若干开放问题,读本暂不回答,记录如下:

- **共享只读上下文机制**:两个子 agent 都需要同一份参考资料(比如同一份规范文档),各自独立读一遍是纯浪费。Anthropic API 当前没有"多 agent 共享只读前缀"的原生机制,只能靠 prompt cache 间接省一部分。这个空白未来会不会被填?→ 观察 2026-2027 年 API 演进
- **子 agent 的原始上下文回溯**:主 agent 事后想追问"你刚才读到的第 27 个文件第 134 行注释是什么",做不到——子 agent 已销毁。能不能做"子 agent 上下文快照",让主 agent 按需查询?成本模型是什么?→ 待产品侧探索
- **"agent 数量 vs 质量非线性"拐点**:Anthropic 观察到过某点就浪费,但那个点具体在哪?跟任务类型、模型规模有没有关系?目前没有公开的定量研究。→ 待 HCI / benchmark 论文
- **跨 agent 的失败归因**:当最终结果差,到底是主 agent 综合错了、子 agent 执行错了、还是派活 prompt 写错了?目前没有标准的 debug 工具链。→ multi-agent 可观测性空白
- **Agent 自主性与 HITL 的最佳分界**:什么风险等级的任务必须在主 agent 决定派子 agent 之前就让人确认?见 [[Wiki/Concepts/AIFoundations/Ai-agent]] 开放问题

## 下一步预告

本读本是 AI 应用生态的一个**支线深度读本**,下一步可深入的方向:

- **Harness Engineering 视角的多 agent**:多 agent 只是 Harness 四大支柱中"上下文管理"的**一个工具**,Harness 还有约束、反馈回路、熵管理三个支柱。→ 见 [[Wiki/Concepts/AIFoundations/Harness-engineering]],以及 [[Wiki/Syntheses/AIFoundations/Prompt-context-harness-evolution]]
- **Agent Skills 的协同**:Skills 是另一个对抗 Context Rot 的机制(渐进式披露),和 Subagent 是互补关系。一个 agent 可能既挂着 Skills,又可以派子 agent——两种机制的组合设计学问还没人系统讲过。→ 见 [[Wiki/Concepts/AIFoundations/Agent-skills]]
- **MCP 与多 agent 的工具分配**:MCP 让工具可被标准化地接入,多 agent 架构下每个子 agent 只需要一部分 MCP server——这里的权限/配置管理还是相对粗糙的。→ 见 [[Wiki/Concepts/AIFoundations/Mcp]]
